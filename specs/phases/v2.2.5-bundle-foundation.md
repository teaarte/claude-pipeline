# Phase v2.2.5 — Bundle foundation + Classification activation

**Status:** shipped via PR #4 (`e3eb3d6`) on 2026-05-18. Items 1-11 all landed; followups bundle (`v2.2.5-followups.md`) merged in the same PR.
**Estimated effort:** ~7-9 days focused work.
**Prerequisite:** v2.2a merged (`bf39b09`).

## Why this phase exists

Pipeline today is architecturally domain-agnostic at the core (FSM, plugin contracts, MCP enforcement, audit) but **coupled to code domain** at four layers: state schema (Phase enum, TestsMode, stack), built-in plugins (all 23 agents + 6 decisions + 3 flows are code-specific), skills (`commands/task.md` references git diff / lint / build), and the planned Web UI (`ui-vision.md` screens show code-review concepts).

The end-goal vision (see [`../product-vision.md`](../product-vision.md), [`../ui-vision.md`](../ui-vision.md)) is **virtual teams for any niche**: code, content generation, marketing, research, VFX, etc. The `Bundle` abstraction makes this real.

**Why now:** v2.3 daemon + Web UI is the next big phase. Building it code-only and then retrofitting bundle-awareness later means rewriting React components. This phase locks in the substrate cheaply (~5-7d) so v2.3+ ships bundle-aware from day 1.

**Closes** Q40 (domain bundle abstraction) — it stops being "deferred for trigger", becomes shipped foundation.

`BundleManifest` interface (in `mcp/src/driver/types/bundle.ts` — new file):
```typescript
export interface BundleManifest extends PluginMeta {
  name: string;                          // e.g. "code", "tiktok", "marketing"
  version: string;                       // semver
  description: string;
  default_flow: string;                  // which flow runs when complexity unset
  supported_flows: string[];             // which flow names this bundle ships
  supported_decisions: string[];         // decision plugin names

  supported_steps: string[];             // step plugin names
  supported_hooks: string[];             // hook plugin names
  supported_gates: string[];             // gate plugin names
  task_prompt_template_path: string;     // relative path to bundle's task preamble
  knowledge_dir?: string;                // relative path to shared knowledge dir
  state_schema_extension?: string;       // path to JSON schema fragment for bundle-specific state fields
}
```

### 2. Phase becomes flow-declared (not global enum)

Today: `Phase = "context" | "planning" | "test_first" | "implementation" | "validation" | "final"` — hard-coded enum in `lib/phase-state-machine.ts`.

After: `Phase = string` (runtime-validated). FlowPlugin declares its phases:
```typescript
export interface FlowPlugin extends PluginMeta {
  name: string;
  complexity: string;
  phases: string[];        // NEW: ordered list of phase names
  steps: string[];         // existing: ordered step names
}
```

Code-bundle flows declare current 6 phases. A future TikTok bundle's flow could declare `["context", "planning", "draft", "critique", "render", "publish"]`.

`pipeline-state.schema.json` validates `phases.<phase>` keys against flow's declared list (ajv `additionalProperties: false` per flow, OR runtime check in `pipeline_validate`).

### 3. State schema bundle-aware

`pipeline-state.json` gains:
```json
{
  "schema_version": "1.1",
  "bundle": "code",                ← NEW required field
  ...
}
```

Schema split into:
- `pipeline-state.schema.json` — base shape (universal: task_id, complexity, verdict, gates, reviewer_verdicts, phases, findings_path)
- `bundles/code/state-extension.schema.json` — code-specific fields (`tests_mode`, `stack`, `files`)
- Future bundles ship their own state-extension schemas

ajv conditional validation: when `state.bundle === "code"`, apply code extension; otherwise apply that bundle's extension.

`schema_version` bumps to `"1.1"` — additive change (new required `bundle` field with default "code" filled in by `pipeline_init`). Old v2.x state files auto-migrate by reading `bundle: "code"` default.

### 4. Loader bundle-aware

`mcp/src/driver/loaders/builtins.ts` → `mcp/src/driver/loaders/bundles.ts`:
```typescript
export async function loadBundle(name: string, registry: PluginRegistry): Promise<BundleManifest> {
  const bundleDir = path.join(__dirname, "..", "bundles", name);
  const manifest = await import(`${bundleDir}/bundle.ts`);
  // register all plugins from manifest
  // return manifest for downstream config consumers
}

export async function loadBundles(names: string[], registry: PluginRegistry): Promise<BundleManifest[]> {
  return Promise.all(names.map(n => loadBundle(n, registry)));
}
```

Project config (`<project>/.claude/pipeline.config.json` — NEW):
```json
{
  "bundle": "code",                    // which bundle to use; default if missing
  "mcp_clients": [],                   // see below
  "team_knowledge_refs": []            // see below
}
```

If `pipeline.config.json` missing → defaults to `{ "bundle": "code" }`. Existing projects work unchanged.

### 6. External MCP consumer pattern (NEW capability)

New plugin contract: `MCPClientPlugin`. Pipeline becomes both MCP server AND MCP client.

```typescript
export interface MCPClientPlugin extends PluginMeta {
  name: string;                                // identifier
  server_command: string[];                    // e.g. ["npx", "claude-mem", "mcp-server"]
  env?: Record<string, string>;                // env vars for the server process
  expose_tools: string[];                      // which tools to make available to agents
  scope: "task" | "team" | "global";          // lifecycle scope
  health_check?: { tool: string; timeout_ms: number };
}
```

Implementation in `mcp/src/driver/builtin/mcp-clients/`:
- Daemon spawns the external MCP server process at startup (or task-start, depending on scope)
- MCP handshake; cache available tools
- Expose declared tools to AgentPlugins via a new field `AgentPlugin.mcp_tools?: string[]` (which tools the agent can call)
- Lifecycle: stop external server on `pipeline_finish` for `scope: "task"`; keep running for `team`/`global`

This unlocks:
- `claude-mem` integration via config (no code change in pipeline)
- Future MCP servers (search, github, etc.) plug in declaratively
- v2.6 marketplace can ship "TikTok bundle bundled with `tiktok-trending-mcp`"

`pipeline.config.json` extension:
```json
{
  "bundle": "code",
  "mcp_clients": [
    {
      "name": "claude-mem",
      "server_command": ["npx", "claude-mem", "mcp-server"],
      "expose_tools": ["search", "timeline", "get_observations"],
      "scope": "team"
    }
  ]
}
```

### 7. Team-scoped knowledge primitive

`state.team_knowledge_refs: string[]` — paths to markdown files this team has accumulated as shared knowledge across tasks.

Bundle ships baseline knowledge in `bundles/<name>/knowledge/`:
- `bundles/code/knowledge/` — empty for now (per-project knowledge accumulates here over time)

Per-task: pipeline reads `team_knowledge_refs[]`, injects relevant ones into agent prompts (similar to refs-to-load but team-scoped, not stack-scoped).

**Write API for shared knowledge** is v2.6 curator territory (curator decides what's worth promoting from per-task analyzer-claims to team knowledge base). v2.2.5 just lays the slot: `state.team_knowledge_refs` field, path resolution helper, agent prompt injection.

### 8. Structured gate-answer protocol (closes Q57)

Free-text gate answer (`"approve"`/`"reject with feedback: ..."`/`"да"`) replaced with structured shape. Category 2 fix per architectural principle — eliminates classification rather than patching it with multilingual keyword lists.

**New type** in `mcp/src/driver/types/plugin.ts`:
```typescript
type UserAnswer = {
  decision: "accept" | "reject";
  message?: string;
};
```

**`pipeline_continue_task input`** schema bumps to accept the new shape:
```typescript
{
  type: "user-answer",
  decision: "accept" | "reject",
  message?: string,
}
```
Backward-compat: also accept legacy `{type: "user-answer", answer: string}` for 2 weeks post-merge, then remove.

**Gate mirror** in `mcp/src/driver/builtin/gates/index.ts` collapses from regex-classifier to one ternary:
```typescript
function parseDecision(input: UserAnswer): GateResult {
  return {
    status: input.decision === "accept" ? "approved" : "rejected",
    feedback: input.message ?? null,
  };
}
```

**Skill markdown** (`commands/task.md`, `commands/done.md`) gains a 5-line parser to accept CLI shorthand:
- `1` / `a` / `accept` → `{decision: "accept"}`
- `2` / `r` / `reject [message]` → `{decision: "reject", message?}`

Web UI / Telegram / API channels (v2.3+) emit the structured shape directly — no parser needed at those channels.

**Tests:** 4 mirror tests (accept / accept+message / reject / reject+message), 4 skill-parser tests, 2 backward-compat tests (legacy free-text still works during transition).

### 9. Classifier-agent + `pickFromCandidates` primitive (closes Q41 / Q44 / Q46 / Q58 / Q61)

Single classifier-agent in `context` phase replaces multiple per-decision LLM calls. Uses existing shuttle infrastructure — no daemon required.

**New primitive** `mcp/src/lib/pick-from-candidates.ts` (~55 lines):
```typescript
export async function pickFromCandidates<T extends string>(args: {
  query: SpawnQuery | undefined;
  task: string;
  candidates: T[];
  cap: number;
  context_hint?: string;
}): Promise<T[]>;
```
Pure helper: prompt assembly + defensive JSON-array parse + filter against `known` Set + cap. Used by classifier-agent today (via Impl A shuttle) and by future `query()` callers (Impl B daemon). One file, one source of truth.

**New agent** `agents/classifier.md` — haiku model, runs in context phase after `code-analyzer` + `context-doc-verifier`. Prompt structure:
- Inputs: task description, CLAUDE.md, refs catalog (with frontmatter), CLAUDE.md anti-pattern rules
- Output: structured JSON with all classification decisions

Output schema:
```json
{
  "task_short": "doc-drift-fix",
  "refs_to_load": ["security-backend.md", "api-design.md"],
  "security_needed": true,
  "antipattern_rules_applicable": ["rule-id-3", "rule-id-7"]
}
```

**New step** `bundles/code/steps/classify.ts` spawns the classifier-agent (shuttle), receives output, validates against schema, writes to `state.decisions.{...}` and `state.task_short`.

**Decisions become pure getters.** Each ~3-5 line:
```typescript
// security-needed.ts post-migration
decide(state) { return state.decisions.security_needed ?? false; }

// refs-to-load.ts post-migration
decide(state) { return state.decisions.refs_to_load ?? []; }
```

**Fallback** when classifier-agent fails (LLM unavailable, malformed output, schema mismatch): decisions return defaults (empty array / false / null task_short). Audit emits `error_class: "llm-classification-needed"`.

**Anti-pattern hook** (`anti-pattern-grep`) reads `state.decisions.antipattern_rules_applicable` and writes only the applicable rule entries to `antipattern-candidates.md`. Word-overlap heuristic deleted.

**Closes:** Q41 (refs LLM-path now active), Q44 (anti-pattern false-positives gone), Q45 (subsumed by Q41), Q46 (`task_short` from LLM), Q58 (security_needed via classifier), Q61 (primitive extracted).

**Does NOT close** Q59 (anti-pattern section header marker convention) — that's a separate Category 2 fix. Defer to Item 10.

## What NOT to do in this phase

❌ **No new domain bundles.** The `_template/` skeleton is documentation only. Building actual `tiktok/` or `marketing/` bundles is a separate phase when validated demand exists.

❌ **No write API for team knowledge.** v2.6 curator decides what gets promoted. v2.2.5 just lays the slot.

❌ **No `pipeline.config.json` editor UI.** That's v2.3 Web UI. v2.2.5 reads the file if present, defaults otherwise.

❌ **No backward-incompat schema changes beyond `bundle` field.** Everything else stays additive.

❌ **No MCP marketplace.** External MCP integration is config-level only — daemon spawns the process, that's it. Curated MCP catalog is v2.6.

## Impact on subsequent phases

- **v2.3 Daemon + Web UI** — builds bundle-aware UI from day 1. Web UI screens accept `bundle: string`. Agent builder lists agents filtered by active bundle. Specialist composer pulls from bundle's plugin catalog.
- **v2.4 Docker** — bundle determines which docker image to build (each bundle can ship its own image with bundle-specific tooling).
- **v2.5 Multi-provider routing** — bundle-aware model preferences (e.g., code bundle prefers opus for planner; future creative bundle might prefer sonnet for creative tasks).
- **v2.6 Marketplace + Curator** — marketplace ships **bundles** (complete domain implementations), not just plugins. Curator becomes bundle-aware.
- **v3.0 Fleet** — multi-bundle teams natively supported. One workspace can have a code-team and a content-team and a research-team, each with own curator.
