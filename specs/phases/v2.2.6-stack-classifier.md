## Why this phase exists

The user spotted an architectural inconsistency during the post-v2.2.5 review:

> "Стэк детектится кодом и тулзы для тестирования детектятся кодом. Если появится проект на C# или Svelte — пайплайн будет корректно работать? Моя изначальная задумка чтобы я просто докидывал референсы. У нас сильная зависимость от CLAUDE.md."

The intuition is correct, and the documented doctrine already supports it (`product-vision.md:203-260` — "Category 3: code + LLM picking from candidate list"). But the doctrine carves out an exception at line 257 — stack-detect is classified as Category 1 ("deterministic"). On reflection this is wrong: "which language is this project?" / "which test runner?" / "which lint command?" are textbook classification problems — the candidate set is finite and the input is a mix of structural signals (lockfile presence) and prose (CLAUDE.md sections / package.json scripts). Hard-coding per-language detection in TypeScript:

- breaks for ecosystems we haven't anticipated (C# .NET, Svelte/SolidJS/Astro/Remix, Elixir/Erlang/Gleam, Dart Flutter outside the default conventions, Zig, Crystal, …)
- forces a TS code edit + release for every new ecosystem
- creates a hard dependency on CLAUDE.md "Validation Commands" prose for non-default test runners
- locks the pipeline to Anthropic models that reliably parse CLAUDE.md — other providers (esp. local/Ollama models) may not pick up the file

The fix is **table-driven candidates + classifier-consumption**: a single YAML file lists known languages / package managers / command shapes; classifier-agent picks from those candidates when it runs; today's path uses the same YAML as a deterministic fallback so we don't gate the fix on classifier auto-spawn.

### Item 2 — `stack-detect.ts` rewrite as candidate-driven

Replace per-language hardcoded paths with two functions:

```typescript
// 1) Code part (Category 1): scan project directory, extract signals.
async function gatherStackSignals(projectDir: string): Promise<StackSignals> {
  return {
    files_present: [...],          // basenames matching candidate signal_files
    package_json: <parsed or null>,
    claude_md_validation_block: <parsed or null>,  // see Item 3
    file_extension_counts: { ".ts": 42, ".rs": 3, ... },
  };
}

// 2) Resolution part (Category 1 today, Category 3 when classifier auto-spawns):
async function resolveStack(signals: StackSignals, candidates: Candidates): Promise<DetectedStack> {
  // Today: deterministic match — pick the language with strongest signal
  // (signal_files present + extension counts), pick the PM whose signal_files match,
  // pick command shape from default_commands[language][pm], with CLAUDE.md overrides.
  // No language-specific code branches.
}
```

After the rewrite:

- ~200 lines of per-language regex go away.
- Adding C#/Svelte/Elixir support = edit `stack-candidates.yaml`, no TS change.
- The Q50/Q51 fixes from v2.2.5-followups stay (they applied to the now-deleted code; their tests move to the new candidate-driven flow).

### Item 3 — Classifier-agent extends to emit stack

`agents/classifier.md` gains a new output section. `templates/schemas/classifier-output.schema.json` adds:

```json
"stack": {
  "type": "object",
  "required": ["language", "package_manager"],
  "properties": {
    "language": { "type": "string" },
    "package_manager": { "type": ["string", "null"] },
    "test_command": { "type": ["string", "null"] },
    "lint_command": { "type": ["string", "null"] },
    "build_command": { "type": ["string", "null"] },
    "project_type": { "type": ["string", "null"], "enum": [null, "frontend-app", "backend", "library", "monorepo"] }
  },
  "additionalProperties": true
}
```

Classifier prompt receives:
- The gathered `StackSignals` (files present, package.json excerpt, etc.)
- The candidate lists from `stack-candidates.yaml`

It picks from candidates. Output goes into `state.stack` (and crucially: when classifier runs, its picks **override** the deterministic `resolveStack` defaults — same pattern as refs_to_load).

Behavior:
- YAML-driven deterministic path runs at `pipelineInit`, populates `state.stack` as a baseline. C#/Svelte/Elixir work because they're in the YAML.
- Classifier-agent auto-spawn (Item 6 below — added to scope after frontend-core run) overrides the table with LLM picks when it runs; deterministic path is the fallback when classifier fails or is unavailable.

### Item 4 — CLAUDE.md hard-dep reduction

Three concrete changes:

1. **Marker convention for validation commands** — parallel to `<!-- antipattern -->`:
   ```markdown
   <!-- validation-commands -->
   - test: pnpm -r test
   - lint: pnpm lint
   - build: pnpm build
   <!-- /validation-commands -->
   ```
   Parser tries the marker block first, falls back to the existing "## Validation Commands" English header. Language-agnostic.

2. **CLAUDE.md becomes optional, not implied-required.** When CLAUDE.md is absent OR the validation block is missing, the classifier (or deterministic fallback) picks defaults from `stack-candidates.yaml`. Pipeline never fails because CLAUDE.md isn't authored.

3. **`mcp/README.md` "Authoring CLAUDE.md" section** updated to document the marker conventions (`<!-- antipattern -->`, `<!-- validation-commands -->`) as the canonical forms; English headers explicitly marked as backward-compat.

Net effect: a project authored entirely by an LLM without human curation (no CLAUDE.md at all) still gets reasonable stack detection. A project with CLAUDE.md gets even better results. Both work.

### Item 6 — Canonical task_id propagation fix

**Symptom:** on frontend-core, findings emitted by challenger / style / performance / acceptance / impl-phase logic-reviewer carried `task_id: "phase-0.7-step-1"` instead of the canonical `t-2026-05-18-implementphase07step`. Agents picked the semantic id from the task description's first line instead of the pipeline-assigned `state.task_id`.

**Consequence:** every analytics query `SELECT * FROM findings WHERE task_id = ?` on the canonical id returns zero rows from that task. Audit-trail story (v2.3 dashboards, v2.6 curator escalation) silently breaks.

**Fix:**

1. `bundles/code/spawn/shuttle-provider.ts buildPrompt` already includes spawn-context metadata. Add an explicit dedicated section that's hard to miss:
   ```markdown
   ## Canonical identifiers (MUST use these in your output JSON)

   - `task_id`: <state.task_id>
   - `driver_state_id`: <state.driver_state_id>
   ```
   Place this section BEFORE the role template injection so agents see it as authoritative project context, not afterthought.

2. Update every reviewer / validator / nonreview agent template (`agents/{logic-reviewer,challenger-reviewer,style-reviewer,security,performance,acceptance,api-contract,...}.md`) — existing "Output constraints" / "Output format" sections gain one line:
   > Your output JSON's `task_id` field MUST equal the canonical `task_id` from the spawn context's "Canonical identifiers" section. Do NOT extract a task_id from the task description prose.

3. Add a defensive runtime check: `pipeline_record_agent_run` (and `record_nonreview_agent`) compare the agent-emitted `task_id` to `state.task_id`. On mismatch:
   - If the emitted id matches a known format-different surface form (semantic vs canonical) → overwrite to canonical + audit warning `task_id-rewrite`.
   - If completely unrelated → reject the record with a clear error message naming the discrepancy.

**Test:** synthetic agent output with `task_id: "wrong-id"` → record handler overwrites to canonical (with audit), or rejects, depending on similarity heuristic. Regression test on the frontend-core fixture data.

**Effort:** ~3-4h. Mostly mechanical template edits + the defensive runtime check.

### Item 7 — Auto-close validation + final phases on clean success (Q63)

**Symptom:** frontend-core 2026-05-18 task closed with `verdict: "accepted"` but the metric row carries `force_used: true` + `pipeline_violation: "phase-force-final"`. This is the **same workflow shape as Q54** (force-final after gate-2 rejection-with-feedback), but now happening on a clean success path. Every successful run will produce this marker.

**Root cause:**
- After `acceptance: PASS` reviewer-record, the `validation` phase stays `in_progress` — acceptance recorded but no one explicitly closes the phase.
- `final` stays `pending` from init; nothing transitions it.
- `/done` calls `pipeline_validate` → INV_007 fires (`verdict` can't be `accepted` while validation/final are not in `{completed, skipped}`).
- User has to `pipeline_set_phase_status final completed force=true` → writes `pipeline_violation: "phase-force-final"`.
- Then `pipeline_finish verdict=accepted` succeeds, but the violation marker is permanently in the metric row.

**Consequence:** `force_used` and `pipeline_violation` (the two columns just added in v2.2.5 Item 11 to surface genuine recovery events) silently fire on EVERY clean run. Cross-project analytics "what percentage of runs needed force?" returns ~100%. The signal is destroyed.

**Fix (two surgical changes):**

1. **Auto-close `validation` on `acceptance: PASS`** — in `mcp/src/tools/record-agent-run.ts` (or the post-record hook path), after recording a reviewer-output with `agent: "acceptance"` AND `verdict: "PASS"` AND `phase: "validation"` AND `open_spawns: 0`: transition `phases.validation` to `completed` without `force=true`. Audit row `auto-close-validation`. Skipped if the agent verdict is NOT PASS — then validation stays in_progress and the user resolves manually.

2. **Auto-close `final` on clean success in `pipeline_finish`** — in `mcp/src/tools/finish.ts`, after the invariant check passes AND `verdict === "accepted"` AND all prior phases are in `{completed, skipped}` AND `phases.final.status === "pending"`: transition `final` to `completed` directly (no `force=true`). No `pipeline_violation` written.

The `force=true` path remains intact for genuine recovery (Q54: gate-2 rejection-then-ship) — that flow continues to write the violation marker as designed.

**Test:**
- Synthetic state with acceptance:PASS + validation in_progress → `record_agent_run` auto-closes validation.
- Synthetic state with verdict=accepted + final=pending + clean prior phases → `pipeline_finish` auto-closes final, no pipeline_violation written.
- Regression: synthetic state with acceptance:FAIL → validation stays in_progress (no auto-close).
- Regression: synthetic state with force_used=true at prior force-set-phase → pipeline_violation preserved through finish.

**Effort:** ~3-4h. Two small handler additions + four tests.

### Item 8 — Cross-session ownership (Q64): hook only blocks the OWNER session

**Symptom (real-task 2026-05-18 18:42):** user had Claude Code window A running `/task` for Phase 0.7 Step 2. Window B in the SAME project was holding an unrelated side-conversation. When window B tried to close, `hooks/pipeline-stop.sh` saw `.claude/pipeline-state.json` with `verdict: null` from window A's task and refused to let B stop:

> "Pipeline is in flight (flow=medium step=5) with verdict=null. Run /done to finalize the task…"

Window B literally cannot close cleanly until window A's `/done`. If window B (or an LLM acting in it) misreads the message and runs `/done` or `pipeline_abandon`, it **kills window A's state**. Pipeline silently treats both sessions as the same owner.

**Root cause:** pipeline's mental model is "one project_dir = one active task". Multi-window workflow in the same project breaks the assumption.

**Fix (designed to avoid Claude-Code binding):**

1. **New generic field `state.owner_id: string | null`** in the base pipeline-state schema. Platform-agnostic — any stable string identifying the entity that started the task. Set on `pipelineInit` / `pipelineRunTask`.

2. **`pipeline_run_task` populates it from a generic env-var chain** at MCP-server-process start:
   ```typescript
   const ownerId =
     process.env.CLAUDE_PIPELINE_OWNER_ID ||
     process.env.CLAUDE_SESSION_ID ||
     process.env.SESSION_ID ||
     null;
   state.owner_id = ownerId;
   ```
   Each integration (Claude Code today, future HTTP API, daemon, CLI) sets the appropriate env var BEFORE spawning the MCP server. Pipeline itself never knows about CC specifically.

3. **`hooks/pipeline-stop.sh` (CC-specific by definition) gains owner comparison:**
   - Read CC session_id from stdin JSON.
   - Read `state.owner_id` from `.claude/pipeline-state.json`.
   - If match → current behavior (block stop, suggest `/done`).
   - If mismatch → INFO message + allow stop with explicit "Do NOT run /done in this session".
   - If `owner_id` is null → fall back to current behavior + note about missing field.

4. **`pipeline_finish` and `pipeline_abandon` gain owner check** with `OWNER_MISMATCH` error code. Override via explicit `force_cross_owner: true` parameter (audited as `pipeline_violation: "cross-owner-finalize"`).

5. **Tests:** synthetic mismatched session ids → tool refuses cleanly. Stop hook test (bash) — mismatched sessions → info exit code 0, not blocking. Regression: matching session_id still blocks.

**Why this design avoids Claude-Code binding:**

- The pipeline doesn't know what `owner_id` means — just a stable opaque string.
- CC integration translates `$CLAUDE_SESSION_ID` → env var → state field. CC-specific code lives only in the integration layer.
- v2.3 daemon will use an HTTP-request-derived owner (auth token / session cookie / user_id), no Claude Code involvement.
- Future CLI / Ollama integrations: owner_id is whatever the integration sets.

The Stop hook itself IS CC-specific (it's a CC plugin event) — but its existence doesn't bind anything else.

**Effort:** ~4h. Schema field + env-var read + hook update + 2 MCP tool checks + 5-6 tests.

## Risk register

- **YAML load latency** — file is small (~5KB), parsed once per process. Negligible. Cache in module-singleton like vocab.
- **YAML authorship errors** — Zod schema (or hand-rolled validator) on load rejects malformed entries with clear errors. Bad YAML fails fast at `loadBundle("code", ...)`.
- **`stack-detect.ts` behavior regression** — comprehensive test fixtures (existing Node + pnpm + Python + Rust + Go cases stay; add C# + Svelte + Elixir + Dart). Snapshot comparisons before/after the rewrite.
- **Classifier schema extension reaches production unused** — fine. Forward-compatible by design. The classifier-agent template change is the only user-visible effect, and only when classifier eventually runs.
- **CLAUDE.md marker convention adoption** — gradual. English-header fallback stays. No project breaks because they didn't switch.
- **Doctrine update creates obligation to revisit other Category 1 sites** — yes, deliberately. Comment in product-vision.md should say "if you find another 'deterministic' classification, audit it the same way".
- **task_id fix (Item 6) defensive rewrite is sneaky** — silently rewriting an agent's emitted `task_id` to canonical could mask bugs where the agent really did intend a different id. Mitigation: ALWAYS audit `task_id-rewrite` with both values; if mismatch ratio exceeds 10% of runs, surface as a metric for investigation.
- **Q63 auto-close (Item 7) might hide failed validation** — auto-closing validation on acceptance:PASS is fine; but if `acceptance` itself emits a malformed verdict that gets parsed as PASS, validation closes silently. Mitigation: strict enum check on verdict value before triggering auto-close; reject anything that's not exactly the string "PASS".
- **Q64 owner_id leak via shared env** — if multiple Claude Code instances share an env (rare but possible — e.g. inside the same shell session), they'd see the same `CLAUDE_SESSION_ID` and pipeline would treat them as one owner. Mitigation: this is the actual SHARED-OWNER case (single user, one window-set) — current behavior is correct; the bug is only when distinct sessions share a project_dir, which Claude Code distinguishes via different session_ids.
