# Phase v2.2.6 — Stack-classifier + CLAUDE.md hard-dep reduction + post-frontend-core fixes

**Status:** scheduled.
**Prerequisite:** v2.2.5-bundle-foundation + v2.2.5-followups merged.
**Estimated effort:** ~4.5-5.5 days (originally ~2 days for stack-classifier; expanded after real-task validation run on frontend-core 2026-05-18 surfaced five additional priorities that fit naturally into the same architectural shift — Items 6-10).

## Why this phase exists

The user spotted an architectural inconsistency during the post-v2.2.5 review:

> "Стэк детектится кодом и тулзы для тестирования детектятся кодом. Если появится проект на C# или Svelte — пайплайн будет корректно работать? Моя изначальная задумка чтобы я просто докидывал референсы. У нас сильная зависимость от CLAUDE.md."

The intuition is correct, and the documented doctrine already supports it (`product-vision.md:203-260` — "Category 3: code + LLM picking from candidate list"). But the doctrine carves out an exception at line 257 — stack-detect is classified as Category 1 ("deterministic"). On reflection this is wrong: "which language is this project?" / "which test runner?" / "which lint command?" are textbook classification problems — the candidate set is finite and the input is a mix of structural signals (lockfile presence) and prose (CLAUDE.md sections / package.json scripts). Hard-coding per-language detection in TypeScript:

- breaks for ecosystems we haven't anticipated (C# .NET, Svelte/SolidJS/Astro/Remix, Elixir/Erlang/Gleam, Dart Flutter outside the default conventions, Zig, Crystal, …)
- forces a TS code edit + release for every new ecosystem
- creates a hard dependency on CLAUDE.md "Validation Commands" prose for non-default test runners
- locks the pipeline to Anthropic models that reliably parse CLAUDE.md — other providers (esp. local/Ollama models) may not pick up the file

The fix is **table-driven candidates + classifier-consumption**: a single YAML file lists known languages / package managers / command shapes; classifier-agent picks from those candidates when it runs; today's path uses the same YAML as a deterministic fallback so we don't gate the fix on classifier auto-spawn.

### Item 2 — `stack-detect.ts` rewrite as candidate-driven

Replace per-language hardcoded paths with two functions:

```typescript
// 1) Code part (Category 1): scan project directory, extract signals.
async function gatherStackSignals(projectDir: string): Promise<StackSignals> {
  return {
    files_present: [...],          // basenames matching candidate signal_files
    package_json: <parsed or null>,
    claude_md_validation_block: <parsed or null>,  // see Item 3
    file_extension_counts: { ".ts": 42, ".rs": 3, ... },
  };
}

// 2) Resolution part (Category 1 today, Category 3 when classifier auto-spawns):
async function resolveStack(signals: StackSignals, candidates: Candidates): Promise<DetectedStack> {
  // Today: deterministic match — pick the language with strongest signal
  // (signal_files present + extension counts), pick the PM whose signal_files match,
  // pick command shape from default_commands[language][pm], with CLAUDE.md overrides.
  // No language-specific code branches.
}
```

After the rewrite:

- ~200 lines of per-language regex go away.
- Adding C#/Svelte/Elixir support = edit `stack-candidates.yaml`, no TS change.
- The Q50/Q51 fixes from v2.2.5-followups stay (they applied to the now-deleted code; their tests move to the new candidate-driven flow).

### Item 3 — Classifier-agent extends to emit stack

`agents/classifier.md` gains a new output section. `templates/schemas/classifier-output.schema.json` adds:

```json
"stack": {
  "type": "object",
  "required": ["language", "package_manager"],
  "properties": {
    "language": { "type": "string" },
    "package_manager": { "type": ["string", "null"] },
    "test_command": { "type": ["string", "null"] },
    "lint_command": { "type": ["string", "null"] },
    "build_command": { "type": ["string", "null"] },
    "project_type": { "type": ["string", "null"], "enum": [null, "frontend-app", "backend", "library", "monorepo"] }
  },
  "additionalProperties": true
}
```

Classifier prompt receives:
- The gathered `StackSignals` (files present, package.json excerpt, etc.)
- The candidate lists from `stack-candidates.yaml`

It picks from candidates. Output goes into `state.stack` (and crucially: when classifier runs, its picks **override** the deterministic `resolveStack` defaults — same pattern as refs_to_load).

Behavior:
- YAML-driven deterministic path runs at `pipelineInit`, populates `state.stack` as a baseline. C#/Svelte/Elixir work because they're in the YAML.
- Classifier-agent auto-spawn (Item 6 below — added to scope after frontend-core run) overrides the table with LLM picks when it runs; deterministic path is the fallback when classifier fails or is unavailable.

### Item 4 — CLAUDE.md hard-dep reduction

Three concrete changes:

1. **Marker convention for validation commands** — parallel to `<!-- antipattern -->`:
   ```markdown
   <!-- validation-commands -->
   - test: pnpm -r test
   - lint: pnpm lint
   - build: pnpm build
   <!-- /validation-commands -->
   ```
   Parser tries the marker block first, falls back to the existing "## Validation Commands" English header. Language-agnostic.

2. **CLAUDE.md becomes optional, not implied-required.** When CLAUDE.md is absent OR the validation block is missing, the classifier (or deterministic fallback) picks defaults from `stack-candidates.yaml`. Pipeline never fails because CLAUDE.md isn't authored.

3. **`mcp/README.md` "Authoring CLAUDE.md" section** updated to document the marker conventions (`<!-- antipattern -->`, `<!-- validation-commands -->`) as the canonical forms; English headers explicitly marked as backward-compat.

Net effect: a project authored entirely by an LLM without human curation (no CLAUDE.md at all) still gets reasonable stack detection. A project with CLAUDE.md gets even better results. Both work.

## Real-task observations from frontend-core 2026-05-18

A real /task run on `OSC/frontend-core` (Phase 0.7 Step 1, type-only TS changes across 8 files in a pnpm monorepo) surfaced four issues that fit the same architectural shift as Items 1-5. Adding them to this phase rather than filing as separate followups because they share the classifier-extension thread (Items 6-7 directly extend classifier behavior; Items 8-9 are correctness fixes to make the existing capture loop trustworthy).

### Item 6 — Classifier-agent auto-spawn (was deferred from v2.2.5 Item 9)

**Symptom:** `state.refs_loaded=[]` on the frontend-core run. The 13 ref files in `agents/references/` were never consulted because the classifier-agent doesn't auto-spawn yet. Decisions `refs_to_load` / `security_needed` / `task_short` / `antipattern_rules_applicable` are pure getters from `state.decisions` (per Item 9 substrate), but no upstream populates that state. Net effect: the refs catalog is silently dead on every task.

This was knowingly deferred in v2.2.5 because auto-spawning a classifier in the `context` phase forced rework of ~14 unit tests that drive the FSM without a real LLM (they'd pause waiting on a spawn that never resolves). The deferral was correct then; the frontend-core run shows the cost (a visible product feature is non-functional) is now bigger than the test-rework cost.

**Fix:**

1. New step `bundles/code/steps/classify-agent.ts` (or extend CLASSIFY) that spawns the `classifier` agent in the `context` phase. Uses the existing `spawnOne` machinery; pattern matches Item 7 of v2.2.5 (team-knowledge fan-out integration) — `team_knowledge` AND classifier-input bundled into the spawn prompt.
2. Output handling: parse JSON in `agent-result` shuttle → validate against `classifier-output.schema.json` → populate `state.decisions.{refs_to_load, security_needed, antipattern_rules_applicable, task_short, stack}` (stack added in Item 3 of this phase).
3. Failure mode: if validation fails OR the agent emits no parseable JSON, decisions retain their defaults (`[]`, `false`, `null`). Audit emits `error_class: "llm-classification-needed"` and the pipeline continues — never fails on a classifier hiccup.
4. Test rework: ~14 FSM tests get an injected mock `SpawnProvider` that resolves the classifier spawn with a stub JSON output. This is a one-time tax to clear; tests after this commit can assume classifier runs normally.

**Side effect:** stack-detect (Items 1-3) gets LLM override path activated, not just deterministic table.

**Effort:** ~1.5-2 days (the test rework is the biggest single block; classifier spawn itself is ~3h).

### Item 7 — `change_kind` emission + reviewer selectivity

**Symptom:** frontend-core run spawned 4 impl-phase reviewers (logic, challenger, style, performance). For a **type-only** TS diff, `style` and `performance` contributed 0 findings each — pure overhead. ~2 wasted spawns × ~5K tokens = ~10K tokens of noise per task on the wrong task-type.

**Fix:**

1. Extend `classifier-output.schema.json` with a new field:
   ```json
   "change_kind": {
     "type": "string",
     "enum": ["type-only", "logic", "ui", "perf-sensitive", "security-sensitive", "config-only", "docs-only"]
   }
   ```
   Multi-label allowed (array form) if a diff is e.g. both logic AND security-sensitive — extend to `string[]` if first-pass simple-string proves insufficient. Start with single-label.

2. `agents/classifier.md` prompt instructs how to pick `change_kind` (filename patterns: `.test.ts` / `.md` / configs → mostly docs/config; absence of runtime code emission → type-only; presence of auth/jwt/session/secret strings → security-sensitive; etc.). The classifier-agent has access to the task description + structural signals from Item 1; it picks from the closed enum.

3. Each `AgentPlugin` (or its `applies_to` function — already in the contract per `plugin.ts:125`) gains a `relevant_for_change_kinds?: string[]` field. Example:
   ```typescript
   const STYLE_REVIEWER: AgentPlugin = {
     name: "style-reviewer", ...
     relevant_for_change_kinds: ["logic", "ui", "security-sensitive"],
     // omits "type-only", "docs-only", "config-only" → not spawned for those
   };
   ```

4. Flow step (`REVIEW` in `bundles/code/steps/index.ts`) consults `state.decisions.change_kind` + each AgentPlugin's `relevant_for_change_kinds` before spawning. Skipped reviewers get an audit row `reviewer-skipped` with the reason — visible in metric row + dashboards.

5. Conservative defaults: when classifier doesn't run / change_kind absent → spawn ALL reviewers (current behavior). Selectivity is an LLM-derived optimization, never a silent skip.

**Test:** synthetic state with `change_kind: "type-only"` → `style-reviewer` and `performance` are skipped; logic + challenger still spawn. Smoke test on a type-only diff confirms ~10K tokens saved vs current behavior.

**Effort:** ~1 day (schema extension + plugin field + flow logic + tests).

### Item 8 — Canonical task_id propagation fix

**Symptom:** on frontend-core, findings emitted by challenger / style / performance / acceptance / impl-phase logic-reviewer carried `task_id: "phase-0.7-step-1"` instead of the canonical `t-2026-05-18-implementphase07step`. The agents picked up the semantic id from the task description's first line ("Implement Phase 0.7 Step 1...") instead of the pipeline-assigned `state.task_id`.

**Consequence:** every analytics query `SELECT * FROM findings WHERE task_id = ?` on the canonical id returns zero rows from that task. The audit-trail story (key v2.3 dashboards, v2.6 curator escalation) silently breaks.

**Fix:**

1. `bundles/code/spawn/shuttle-provider.ts buildPrompt` already includes spawn-context metadata. Add an explicit dedicated section that's hard to miss:
   ```markdown
   ## Canonical identifiers (MUST use these in your output JSON)

   - `task_id`: <state.task_id>
   - `driver_state_id`: <state.driver_state_id>
   ```
   Place this section BEFORE the role template injection so agents see it as authoritative project context, not afterthought.

2. Update every reviewer / validator / nonreview agent template (`agents/{logic-reviewer,challenger-reviewer,style-reviewer,security,performance,acceptance,api-contract,...}.md`) — the existing "Output constraints" / "Output format" sections gain one line:
   > Your output JSON's `task_id` field MUST equal the canonical `task_id` from the spawn context's "Canonical identifiers" section. Do NOT extract a task_id from the task description prose.

3. Add a defensive runtime check: `pipeline_record_agent_run` (and `record_nonreview_agent`) compare the agent-emitted `task_id` to `state.task_id`. On mismatch:
   - If the emitted id matches a known format-different surface form (e.g. semantic vs canonical) → overwrite to canonical + audit warning `task_id-rewrite`.
   - If completely unrelated → reject the record with a clear error message naming the discrepancy.

**Test:** synthetic agent output with `task_id: "wrong-id"` → record handler overwrites to canonical (with audit), or rejects, depending on similarity heuristic. Regression test on the frontend-core fixture data.

**Effort:** ~3-4h. Mostly mechanical template edits + the defensive runtime check.

### Item 9 — Implementer tech-debt auto-capture to `issues-found.md`

**Symptom:** frontend-core implementer mentioned in its output prose: *"Pre-existing prettier debt in repo (19 files): mostly .md files plus a few pre-existing TS files; not a regression."* This is a textbook tech-debt observation for `/sweep` workflow. But `.claude/issues-found.md` was never created — the observation lives only in prose buried in driver-state.scratch (which gets cleaned at `/done`). `/done`'s tech-debt-promotion step finds an empty/absent file and skips. **Workflow gap: the data was captured in passing but lost structurally.**

**Fix:**

1. Update `agents/implementer.md` template — add explicit instruction:
   > **Tech-debt and out-of-scope observations.** If during implementation you notice issues that are NOT part of your task scope (pre-existing bugs, dead code, opportunistic improvements, debt the next maintainer should know about), write each one as a bullet to `.claude/issues-found.md` BEFORE your final output JSON. Do NOT mention them in your output prose — the prose is the work-summary; `issues-found.md` is the structured tech-debt feed for `/sweep`.

2. Add a post-implementation hook `extract-tech-debt-from-prose`:
   - Event: `after-agent-result`, agent filter: `implementer`.
   - Reads the implementer's output prose. Greps for paragraph-level signals: `pre-existing`, `out-of-scope`, `not a regression`, `noticed`, `also worth fixing`, `TODO:`, `FIXME:`, etc.
   - For each match, surfaces the matching block + appends to `.claude/issues-found.md` under a `<!-- auto-captured -->` block (so the human + /sweep see them with provenance).
   - Idempotent: if the same paragraph hash already exists, skip.

3. Update `mcp/src/driver/bundles/code/done-prompt.md` step 4 (tech-debt promotion) — it already exists conceptually; explicitly note the auto-capture source so the workflow is documented end-to-end.

**Test:** synthetic implementer output containing "Pre-existing prettier debt in repo: ..." → hook writes the bullet to `issues-found.md` with auto-captured marker; second invocation with same content is idempotent.

**Effort:** ~3-4h. Template edit + hook implementation + 2-3 tests covering pattern matching + idempotency.

### Item 10 — Auto-close validation + final phases on clean success (Q63)

**Symptom:** frontend-core 2026-05-18 task closed with `verdict: "accepted"` but the metric row carries `force_used: true` + `pipeline_violation: "phase-force-final"`. This is the **same workflow shape as Q54** (force-final after gate-2 rejection-with-feedback), but now happening on a clean success path. Every successful run will produce this marker.

**Root cause:**
- After `acceptance: PASS` reviewer-record, the `validation` phase stays `in_progress` — the acceptance validator is recorded but no one explicitly closes the phase.
- `final` stays `pending` from init; nothing transitions it.
- `/done` calls `pipeline_validate` → INV_007 fires (`verdict` can't be `accepted` while validation/final are not in `{completed, skipped}`).
- User has to `pipeline_set_phase_status final completed force=true` → writes `pipeline_violation: "phase-force-final"`.
- Then `pipeline_finish verdict=accepted` succeeds, but the violation marker is now permanently in the metric row.

**Consequence:** `force_used` and `pipeline_violation` (the two columns just added in v2.2.5 Item 11 to surface genuine recovery events) silently fire on EVERY clean run. Cross-project analytics "what percentage of runs needed force?" returns ~100%. The signal is destroyed.

**Fix (two surgical changes):**

1. **Auto-close `validation` on `acceptance: PASS`** — in `mcp/src/tools/record-agent-run.ts` (or the post-record hook path), after recording a reviewer-output with `agent: "acceptance"` AND `verdict: "PASS"` AND `phase: "validation"` AND `open_spawns: 0`: transition `phases.validation` to `completed` without `force=true`. Audit row `auto-close-validation`. Skipped if the agent verdict is NOT PASS — then validation stays in_progress and the user resolves manually.

2. **Auto-close `final` on clean success in `pipeline_finish`** — in `mcp/src/tools/finish.ts`, after the invariant check passes AND `verdict === "accepted"` AND all prior phases are in `{completed, skipped}` AND `phases.final.status === "pending"`: transition `final` to `completed` directly (no `force=true`). No `pipeline_violation` written.

The `force=true` path remains intact for genuine recovery (Q54: gate-2 rejection-then-ship) — that flow continues to write the violation marker as designed.

**Test:**
- Synthetic state with acceptance:PASS + validation in_progress → `record_agent_run` auto-closes validation.
- Synthetic state with verdict=accepted + final=pending + clean prior phases → `pipeline_finish` auto-closes final, no pipeline_violation written.
- Regression: synthetic state with acceptance:FAIL → validation stays in_progress (no auto-close).
- Regression: synthetic state with force_used=true at prior force-set-phase → pipeline_violation preserved through finish.

**Effort:** ~3-4h. Two small handler additions + four tests.

## Risk register

- **YAML load latency** — file is small (~5KB), parsed once per process. Negligible. Cache in module-singleton like vocab.
- **YAML authorship errors** — Zod schema (or hand-rolled validator) on load rejects malformed entries with clear errors. Bad YAML fails fast at `loadBundle("code", ...)`.
- **`stack-detect.ts` behavior regression** — comprehensive test fixtures (existing Node + pnpm + Python + Rust + Go cases stay; add C# + Svelte + Elixir + Dart). Snapshot comparisons before/after the rewrite.
- **Classifier schema extension reaches production unused** — fine. Forward-compatible by design. The classifier-agent template change is the only user-visible effect, and only when classifier eventually runs.
- **CLAUDE.md marker convention adoption** — gradual. English-header fallback stays. No project breaks because they didn't switch.
- **Doctrine update creates obligation to revisit other Category 1 sites** — yes, deliberately. Comment in product-vision.md should say "if you find another 'deterministic' classification, audit it the same way".
- **Classifier auto-spawn (Item 6) FSM-test rework** — biggest single risk. ~14 unit tests will need an injected mock SpawnProvider that returns stub classifier JSON. Mitigation: write a `mockClassifierSpawnProvider()` test helper that returns canned output; tests reuse it. One-time cost; new tests after this commit assume classifier runs.
- **Reviewer selectivity (Item 7) hides real findings** — if `change_kind` is wrong, a skipped reviewer might've caught a bug. Mitigation: (a) conservative default — when `change_kind` is null/absent → spawn ALL (current behavior), (b) audit row `reviewer-skipped` is queryable, (c) when wandr-be or frontend-core re-run shows selectivity skipping the right reviewer, validate against the previous run's findings (no real blocking finding should have come from the skipped reviewer).
- **task_id fix (Item 8) defensive rewrite is sneaky** — silently rewriting an agent's emitted `task_id` to canonical could mask bugs where the agent really did intend a different id. Mitigation: ALWAYS audit `task_id-rewrite` with both values; if mismatch ratio exceeds 10% of runs, surface as a metric for investigation.
- **Tech-debt auto-capture (Item 9) false positives** — the prose-pattern hook might wrongly capture "pre-existing" mentions in non-tech-debt context. Mitigation: idempotency check (don't double-capture the same paragraph); human reviews `issues-found.md` at `/done` anyway before promotion to `<kb>/tech-debt.md`.
