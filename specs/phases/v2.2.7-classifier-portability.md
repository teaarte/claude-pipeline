# Phase v2.2.7 — Classifier auto-spawn + reviewer selectivity + portability foundations

**Status:** scheduled.
**Prerequisite:** v2.2.6 merged. Specifically depends on:
- v2.2.6 Item 4 (`classifier-output.schema.json` already extended with `stack` + `change_kind`) — this phase ACTIVATES that schema.
- v2.2.6 Item 6 (task_id propagation fix) — classifier auto-spawn would otherwise compound the task_id drift.

**Estimated effort:** ~4-5 days (5 items + D6 planning-fanout added 2026-05-18 from user-observed real-task signal).

## Why this phase exists

v2.2.6 ships the stack-classifier substrate + critical workflow fixes from the frontend-core 2026-05-18 run. v2.2.7 turns three remaining levers:

1. **Activate the LLM-classification cluster** that v2.2.5 Item 9 deferred (substrate exists; auto-spawn doesn't yet → refs catalog silently dead on every task). This is the most-watched gap from real-task validation.
2. **Reviewer selectivity by change_kind** — stop spawning style + performance reviewers on type-only TS diffs (~10K tokens wasted per task per frontend-core observation).
3. **Portability foundations** — generic shuttle response shape + configurable project subdirectory. Lays the groundwork for v2.3 daemon and v2.5 multi-provider without binding pipeline core to Claude Code.

Bundling these together because (a) Items 1 + 2 are tightly coupled (selectivity reads `change_kind` from classifier output — needs classifier to actually run), and (b) the portability items (Q65, Q66) are architectural prep that's cheaper to land alongside the classifier-activation churn than as a separate session.

## What ships

### Item 1 — Classifier-agent auto-spawn (was v2.2.6 Item 6, deferred from v2.2.5 Item 9)

**Symptom:** `state.refs_loaded=[]` on every real-task run. The 13 ref files in `agents/references/` were never consulted because the classifier-agent doesn't auto-spawn. Decisions `refs_to_load` / `security_needed` / `task_short` / `antipattern_rules_applicable` are pure getters from `state.decisions` (per v2.2.5 Item 9 substrate), but no upstream populates that state.

This was knowingly deferred in v2.2.5 because auto-spawning a classifier in the `context` phase forced rework of ~14 unit tests that drive the FSM without a real LLM. The deferral was correct then; the frontend-core run shows the cost (a visible product feature is non-functional) is now bigger than the test-rework cost.

**Fix:**

1. New step `bundles/code/steps/classify-agent.ts` (or extend CLASSIFY) that spawns the `classifier` agent in the `context` phase. Uses the existing `spawnOne` machinery; pattern matches v2.2.5 Item 7 (team-knowledge fan-out integration) — `team_knowledge` AND classifier-input bundled into the spawn prompt.
2. Output handling: parse JSON in `agent-result` shuttle → validate against `classifier-output.schema.json` (already extended in v2.2.6 Item 4 with `stack` + `change_kind`) → populate `state.decisions.{refs_to_load, security_needed, antipattern_rules_applicable, task_short, stack, change_kind}`.
3. Failure mode: if validation fails OR the agent emits no parseable JSON, decisions retain their defaults (`[]`, `false`, `null`). Audit emits `error_class: "llm-classification-needed"` and the pipeline continues — never fails on a classifier hiccup.
4. Test rework: ~14 FSM tests get an injected mock `SpawnProvider` that resolves the classifier spawn with a stub JSON output. Helper `mockClassifierSpawnProvider()` written once, reused. One-time tax to clear; tests after this commit can assume classifier runs normally.

**Side effect:** stack-detect from v2.2.6 Items 1-3 gets LLM override path activated (table-driven results become the fallback; LLM picks become the default when classifier ran successfully).

**Effort:** ~1.5-2 days. The test rework is the biggest single block; classifier-spawn itself is ~3h.

### Item 2 — `change_kind` emission + reviewer selectivity (was v2.2.6 Item 7)

**Symptom:** frontend-core run spawned 4 impl-phase reviewers (logic, challenger, style, performance). For a **type-only** TS diff, `style` and `performance` contributed 0 findings each — pure overhead. ~2 wasted spawns × ~5K tokens = ~10K tokens of noise per task on the wrong task-type.

**Fix:**

1. `classifier-output.schema.json` already includes `change_kind` (added in v2.2.6 Item 4). Now the field gets a real consumer:
   ```json
   "change_kind": {
     "type": "string",
     "enum": ["type-only", "logic", "ui", "perf-sensitive", "security-sensitive", "config-only", "docs-only"]
   }
   ```

2. `agents/classifier.md` prompt instructs how to pick `change_kind` (filename patterns: `.test.ts` / `.md` / configs → mostly docs/config; absence of runtime code emission → type-only; presence of auth/jwt/session/secret strings → security-sensitive; etc.).

3. Each `AgentPlugin` gains a `relevant_for_change_kinds?: string[]` field:
   ```typescript
   const STYLE_REVIEWER: AgentPlugin = {
     name: "style-reviewer", ...
     relevant_for_change_kinds: ["logic", "ui", "security-sensitive"],
     // omits "type-only", "docs-only", "config-only" → not spawned for those
   };
   ```

4. Flow step (`REVIEW` in `bundles/code/steps/index.ts`) consults `state.decisions.change_kind` + each AgentPlugin's `relevant_for_change_kinds` before spawning. Skipped reviewers get an audit row `reviewer-skipped` with the reason.

5. Conservative defaults: when classifier-output absent OR `change_kind` is null → spawn ALL reviewers (current behavior). Selectivity is an LLM-derived optimization, never a silent skip.

**Test:** synthetic state with `change_kind: "type-only"` → `style-reviewer` and `performance` are skipped; logic + challenger still spawn. Smoke test on a type-only diff confirms ~10K tokens saved.

**Effort:** ~1 day.

### Item 3 — Implementer tech-debt auto-capture (was v2.2.6 Item 9)

**Symptom:** frontend-core implementer mentioned in its output prose: *"Pre-existing prettier debt in repo (19 files): mostly .md files plus a few pre-existing TS files; not a regression."* This is a textbook tech-debt observation for `/sweep` workflow. But `.claude/issues-found.md` was never created — the observation lives only in driver-state.scratch prose, then gets cleaned at `/done`. **Workflow gap.**

**Fix:**

1. Update `agents/implementer.md` template — explicit instruction:
   > **Tech-debt and out-of-scope observations.** If during implementation you notice issues NOT part of your task scope (pre-existing bugs, dead code, opportunistic improvements, debt the next maintainer should know about), write each one as a bullet to `.claude/issues-found.md` BEFORE your final output JSON. Do NOT mention them in your output prose — the prose is the work-summary; `issues-found.md` is the structured tech-debt feed for `/sweep`.

2. Add a post-implementation hook `extract-tech-debt-from-prose`:
   - Event: `after-agent-result`, agent filter: `implementer`.
   - Reads the implementer's output prose. Greps for paragraph-level signals: `pre-existing`, `out-of-scope`, `not a regression`, `noticed`, `also worth fixing`, `TODO:`, `FIXME:`, etc.
   - For each match, surfaces the matching block + appends to `.claude/issues-found.md` under a `<!-- auto-captured -->` block.
   - Idempotent: if the same paragraph hash already exists, skip.

3. Update `bundles/code/done-prompt.md` step 4 (tech-debt promotion) — explicitly document the auto-capture source.

**Test:** synthetic implementer output containing "Pre-existing prettier debt in repo: ..." → hook writes bullet to `issues-found.md` with auto-captured marker; second invocation idempotent.

**Effort:** ~3-4h.

### Item 4 — Generic shuttle response shape (Q65)

**Symptom:** `mcp/src/driver/types/shuttle.ts` defines `ClaudeCodeTaskSpec` as a top-level type AND `DriverResponse.spawn-agent` carries a `claude_code_task` field. The field NAME and SHAPE are a direct copy of Claude Code's Task-tool parameters. Any non-CC consumer (Cursor MCP integration, future HTTP API client, Codex / Gemini CLI MCP support) sees a field they don't recognize and can't proxy the spawn correctly. This binds pipeline core to a single client at the type level.

**Root cause:** pipeline's spawn-protocol leaked the runner's identity into the type names. Should be runner-agnostic.

**Fix:**

1. Rename + restructure in `mcp/src/driver/types/shuttle.ts`:
   ```typescript
   // BEFORE
   export type ClaudeCodeTaskSpec = {
     subagent_type: string;
     description: string;
     prompt: string;
     model?: "haiku" | "sonnet" | "opus";
   };

   // AFTER
   export type SpawnRequest = {
     runner_hint: "claude-code-task" | "anthropic-sdk" | "openai-sdk" | "ollama" | string;
     // runner-agnostic spec — each runner adapter translates this to its native shape
     description: string;
     prompt: string;
     model?: string;  // any provider-specific id; tier abstraction lives in v2.5 routing
     // Optional runner-specific extras pass through
     extras?: Record<string, unknown>;  // e.g. {"subagent_type": "general-purpose"} for CC
   };

   export type DriverResponse = ... | {
     status: "spawn-agent";
     ...
     spawn_request: SpawnRequest;  // was: claude_code_task: ClaudeCodeTaskSpec
   };
   ```

2. `shuttleSpawnProvider.spawn` returns `runner_hint: "claude-code-task"`, puts `subagent_type` into `extras`. Future `AnthropicSdkSpawnProvider` (v2.3.4) returns `runner_hint: "anthropic-sdk"` with its own extras. The runner-hint is informational for the consumer to know HOW to invoke — pipeline doesn't care.

3. `commands/task.md` skill markdown updates: parses `spawn_request` instead of `claude_code_task`, reads `extras.subagent_type` to feed Claude Code's Task tool. CC-specific translation lives in the SKILL layer, not in pipeline core types.

4. Audit log entry on every spawn includes `runner_hint` — analytics-friendly.

**Why this matters even before v2.3:** Cursor users can already install pipeline as MCP server and invoke `pipeline_run_task`. Today they get a `claude_code_task` field they can't interpret. With `spawn_request.runner_hint`, a Cursor-adapter skill markdown could translate to Cursor's agent-spawn primitive. Same for any other MCP-supporting client.

**Test:** existing shuttle-provider tests stay green after field rename (mechanical edit). New test: assert `runner_hint === "claude-code-task"` + `extras.subagent_type === "general-purpose"` in shuttle output. Backward-compat: NONE (per solo-user no-compat policy — clean cut).

**Effort:** ~1 day. Type rename + provider update + skill markdown + ~5-8 test fixture updates.

### Item 5 — Configurable project subdirectory (Q66)

**Symptom:** `mcp/src/lib/paths.ts` hardcodes `.claude/` as the project's pipeline working directory. This is Claude Code convention. For other transports / clients / future product positioning ("the pipeline isn't a Claude-Code accessory"), this naming creates noise.

**Fix:**

1. Read project subdir from env var with sensible default:
   ```typescript
   export const PROJECT_SUBDIR =
     process.env.CLAUDE_PIPELINE_PROJECT_SUBDIR ||
     ".claude"; // keeps current convention for CC users
   ```

2. `claudeDir(projectDir)` and related path helpers use `PROJECT_SUBDIR` consistently. (Search-replace the hardcoded `.claude` literal in `mcp/src/lib/paths.ts`.)

3. Document the env var in `mcp/README.md`:
   - CC users: leave default (`.claude/`); existing projects keep working.
   - Headless / daemon / non-CC users: set `CLAUDE_PIPELINE_PROJECT_SUBDIR=.pipeline` (or whatever) in env. State writes go to `<project>/.pipeline/` instead.
   - Hook + skill convention adapts (CC's hook still expects `.claude/` — that's CC's concern, not pipeline's).

4. NO migration logic. Per no-backcompat policy, projects either use the default OR explicitly opt into a different subdir. Cross-machine portability is the env-var's job, not pipeline's.

**Why this matters:** when v2.3 daemon ships and users run it without Claude Code, the `.claude/` directory in every project they touch is misleading branding. The env var lets daemon users rename without forking.

**Test:** unit test on `paths.ts` — assert `claudeDir("/proj")` returns `/proj/.claude` by default and `/proj/.pipeline` with env var set. Integration test (skipped in CI by default): run `pipeline_init` with the env var → state writes to alt subdir.

**Effort:** ~2-3h. One file + 5 lines of doc + 2 tests.

### Item 6 — Parallelize planning-phase reviewers (Q67)

**Symptom:** real-task observation 2026-05-18 — `plan-grounding-check` and `logic-reviewer` at planning gate-1 fire as TWO SEPARATE FSM steps:

```
PLAN_GROUNDING.run → spawnOne("plan-grounding-check") → shuttle pause
  → CC runs grounding agent
  → pipeline_continue_task delivers result
  → FSM advances step_index
PLAN_REVIEW.run → spawnOne("logic-reviewer") → shuttle pause
  → CC runs logic agent
  → result delivered
```

Two shuttle round-trips, serialized LLM execution. The implementation phase already uses `spawnAgentsParallel` (Q9 fan-out) for the 5-reviewer batch — the primitive exists in `core/shuttle.ts:32`, just isn't applied at planning.

**Cost:** ~30-60s of wall time per medium/complex task — extra shuttle round-trip + serialized LLM execution where parallel would let CC's Task tool drive both subagents concurrently. With 10+ /task runs/day that's measurable.

**Fix:**

1. Delete the standalone `PLAN_GROUNDING` step. Rewrite `PLAN_REVIEW` step to fan out `[plan-grounding-check, logic-reviewer]` (and optionally `challenger-reviewer` for cross-perspective) in parallel via `spawnAgentsParallel`, mirroring the impl `REVIEW` step pattern (`bundles/code/steps/index.ts:400-472`).
2. SIMPLE flow keeps `plan-grounding` as a standalone step (no fan-out needed; only one agent runs at planning gate-1 in SIMPLE).
3. `flows/index.ts`: drop `plan-grounding` from MEDIUM + COMPLEX step lists. SIMPLE_FLOW keeps it.
4. Resume short-circuit + scratch keys (`__plan_review_agents_issued`) follow the same pattern as `REVIEW_ISSUED_KEY` in impl REVIEW.

**Why this matters:** every medium/complex task pays this overhead. The fix doesn't change pipeline behavior — same agents run, same findings emitted; just concurrent instead of serial. Net wall-time saving is largest on COMPLEX tasks where logic-reviewer takes the longest single span.

**Test:** integration test on a synthetic medium-flow run asserts `pipeline_continue_task` accepts a batched `agents-results` shape carrying both plan-phase reviewers. Existing FSM tests get a one-line update (resume after batched results instead of sequential single-agent shuttles).

**Effort:** ~3-4h. Step merge + flow-list update + FSM-test update.

### Item 7 — `acceptance: PASS` gate on impl-phase blockers (Q68)

**Symptom (frontend-core 2026-05-19, task `t-2026-05-19-implementhybridexecu`):** style-reviewer at impl iteration=2 emitted 3 blocking prettier violations on the new/modified files. Acceptance subsequently returned `verdict: PASS` because its `pnpm test/lint/build` call ran against project scope where 17 pre-existing prettier failures are deliberately excluded — the new-file violations were invisible to the shell command. Without the user's manual catch at Gate 2, the task would have shipped with blocker findings still open.

**Consequence:** the workflow value of the impl-phase reviewer fan-out (Q9) is silently nullified whenever acceptance happens to pass on a stale or scoped baseline. The pipeline's "blocking-issue gate" mental model is broken at the boundary between impl and validation phases.

**Severity: HIGH** — it produces silent ship-with-blockers in real runs. This is the most critical item in v2.2.7 and should land first in the order, before D1's FSM-test rework.

**Fix (two complementary layers):**

1. **Prompt-side instruction in `agents/acceptance.md`.** Add a new section to Output constraints:
   > Before emitting `verdict: PASS`, you MUST query the pipeline state's `reviewer_verdicts[]` (provided in spawn context). If any entry from the implementation phase at the latest iteration has open `blocking_issues > 0`, downgrade your verdict to `FAIL` with a `summary_line` enumerating the blocker categories + file paths from `findings.jsonl`. Do NOT pass purely on tool-call-green if reviewer findings are still open.

2. **Hard invariant `INV_013`.** New entry in `mcp/src/lib/invariants.ts`:
   > acceptance verdict=PASS ⇒ no open `severity: blocking` findings emitted by impl-phase reviewers on the latest iteration.

   Fires from `pipeline_record_agent_run` post-record AND from `pipeline_finish` pre-row-emission. INV_013 makes the prompt-side check load-bearing — even an agent that ignores the prompt instruction can't ship past it.

The data is already there: `findings.jsonl` has `severity`, `agent`, `iteration`, `status: open`; `reviewer_verdicts[]` carries `phase`, `iteration`, `blocking_issues`. Cross-reference is mechanical.

**Test:** synthetic state with `style-reviewer.iteration=2.blocking_issues=3` + acceptance:PASS → INV_013 fires; finish refused. Acceptance prompt update verified via `agents/output-constraints.test.ts`.

**Effort:** ~3-4h. Prompt edit + new invariant + 3-4 tests.

### Item 8 — Auto-derived gate-1 feedback (Q69)

**Symptom:** When logic-reviewer returns `REQUEST_CHANGES` with N findings at planning, the user manually transcribes each `summary + suggested_fix` into the gate-1-reject message body (real-task frontend-core 2026-05-19: 7-bullet revision text manually paraphrasing 5 findings). Findings already carry structured `summary` / `suggested_fix` / `severity` / `category`.

**Fix:**

1. Aggregate latest planning-phase reviewer findings (logic-reviewer + plan-grounding-check) into a pre-filled "Suggested revision" block in `GATE_1.message(state)`. Sort by `severity` desc, then by `agent`, then by `id`. Emit one bullet per finding: `(SEVERITY, category) summary. Suggested fix: <suggested_fix>`.
2. Extend `UserAnswer` schema with a new decision: `"auto-apply"`. The `pipeline_continue_task` shape:
   ```typescript
   { type: "user-answer", decision: "auto-apply" | "accept" | "reject", message?: string }
   ```
3. Skill markdown parser (`commands/task.md`):
   - `1` / `a` / `auto-apply` → `{decision: "auto-apply"}` — re-spawns planner using the auto-derived feedback as the gate-1-reject message
   - `2` / `accept` → today's accept-anyway path
   - `3` / `edit <text>` → text replaces the auto-derived feedback before reject
   - `4` / `reject <msg>` → free-text reject (today's behavior)
4. Mirror logic in `bundles/code/gates/index.ts` `parseDecision` handles the `auto-apply` enum.

**Why this matters:** removes the friction of manually typing 5-7 bullets that match exactly what the findings already say. Doesn't change FSM shape, just enriches the gate prompt + adds the auto-apply path. Sets up D9 (Q70 auto-replan loop).

**Test:** gate-1 message snapshot test confirms the suggested-revision block format; UserAnswer schema test covers the new enum; mirror test exercises auto-apply → reject conversion.

**Effort:** ~4-6h.

### Item 9 — Auto-replan loop on REQUEST_CHANGES (Q70, opt-in)

**Symptom (knock-on):** Item 8's UX reduction is the bottleneck but the loop is still inherently sequential — human stops to read findings each time. For projects where logic-reviewer routinely returns NEEDS_REVISION → auto-apply → APPROVE on a single iteration, the human gate-1 stop is pure friction.

**Fix (opt-in via config):**

1. New config field in `pipeline.config.json`:
   ```json
   { "auto_replan_on_blocking_max": 0 }
   ```
   Default `0` = today's behavior (human Gate 1 every time).
2. When `auto_replan_on_blocking_max >= 1` AND logic-reviewer at planning iteration=N returns REQUEST_CHANGES AND N < max:
   - Use Item 8's auto-derived feedback as a synthetic gate-1-reject message.
   - Skip the gate pause; planner re-spawns automatically with the feedback.
   - Re-run plan-grounding + logic-reviewer fan-out (D6 once landed).
   - If iteration N+1 returns APPROVE → continue to test_first / impl phase normally.
   - If iteration N+1 still REQUEST_CHANGES AND N+1 < max → repeat.
   - If N+1 >= max OR verdict still REQUEST_CHANGES at cap → fall through to manual Gate 1.

**Why capped:** real-task observation 2026-05-19 — gate-1 reject "add `__mfRuntime` seam for AC-8 integration test" prompted planner to add `BootstrapOverrides` DI seam which challenger-reviewer subsequently called "over-engineered" + T-37 "theatre test". Unlimited auto-replan would scale this confirmation-bias pattern. Cap forces human review after N attempts.

**Audit:** each auto-replan iteration writes an audit row `error_class: "auto-replan"` with the synthesized feedback summary. Cross-task analytics can answer "what % of tasks auto-replan converged vs. fell through to manual Gate 1?".

**Depends on Item 8 (Q69)** — needs auto-derived feedback shape.

**Test:** synthetic flow with stubbed logic-reviewer that returns REQUEST_CHANGES once then APPROVE → FSM goes through auto-replan + advances to gate-1-approve without a human pause. Cap test: 3 consecutive REQUEST_CHANGES → fall through to gate-1 manual pause.

**Effort:** ~3-4h.

### Item 10 — Truncate `state.task` in gate messages (Q71)

**Symptom (frontend-core 2026-05-19):** gate-0 and gate-1 message bodies contained the full 10-KB task description verbatim. User had to scroll through their own input to find the `Reply 1/accept or 2/reject` prompt.

**Fix:**

`bundles/code/gates/index.ts`:

```typescript
function shortTask(state: DriverState): string {
  if (state.task_short) return state.task_short;
  const firstLine = state.task.split("\n")[0].trim();
  if (firstLine.length <= 80) return firstLine;
  return firstLine.slice(0, 77) + "...";
}

// GATE_0.message
const complexity = state.decisions["complexity"];
return [
  `Classified as ${String(complexity).toUpperCase()}.`,
  `Task: ${shortTask(state)}`,
  `Reply 1/accept or 2/reject <message>.`,
].join("\n");

// GATE_1.message — same shortTask call, plus the Q69 auto-derived revision block.
```

Prefer `state.task_short` when classifier (D1) populated it; fall back to first-line truncation otherwise. Gate-2 message is already constant — no change needed.

**Test:** snapshot tests on gate-0 + gate-1 messages with 10-KB synthetic `state.task` → assert message stays under 500 chars.

**Effort:** ~1h.

### Item 11 — Documented owner_id env-var propagation + audit warning (Q72)

**Symptom (frontend-core 2026-05-19):** `state.owner_id: null` after `pipelineRunTask` ran in production. None of `CLAUDE_PIPELINE_OWNER_ID` / `CLAUDE_SESSION_ID` / `SESSION_ID` env vars reached the MCP server process. Root cause: CC's `~/.claude.json` mcpServers entry has `"env": {}` and CC does NOT auto-forward its own session_id to stdio-child processes.

**Consequence:** the Q64 cross-session OWNER_MISMATCH check is silently inactive in real production — `ownerCheck()` returns `no-owner-recorded` and the safety pass through. Stop hook still works (it reads session_id from stdin payload), but MCP-level safety on `pipeline_finish` / `pipeline_abandon` is bypassed.

**Fix (two parts, both small):**

1. **`mcp/README.md` install section:** document the required CC config:
   ```jsonc
   // ~/.claude.json
   {
     "mcpServers": {
       "claude-pipeline": {
         "type": "stdio",
         "command": "node",
         "args": ["…/mcp/dist/server.js"],
         "env": {
           "CLAUDE_PIPELINE_OWNER_ID": "${CLAUDE_SESSION_ID}"
         }
       }
     }
   }
   ```
   If CC's env-substitution doesn't trigger (verify against current CC version), document a wrapper-shell fallback.

2. **Audit one-time warning in `pipelineRunTask`:** when `ownerId === null`, emit an `audit({ tool: "pipeline_run_task", error_class: "owner-id-unset", verdict: "ok" })` so the gap is visible in the metrics stream. Per-task overhead: one extra audit row.

**Test:** unit test on `pipelineRunTask` with empty env → audit row `owner-id-unset` emitted; with `CLAUDE_PIPELINE_OWNER_ID=x` → no audit row.

**Effort:** ~2-3h (mostly docs + 1 audit call + 2 tests).

### Item 12 — `summary_line` cap raised or constraint promoted (Q73)

**Symptom (frontend-core 2026-05-19):** 2 schema-validation errors during the planning phase, both `/summary_line: must NOT have more than 100 characters`. Templates explicitly say `summary_line: ≤ 100 chars` in Output constraints, but agents emit longer strings on complex tasks. System self-corrects via lenient-parse retry (`error_class: "schema-validation"`) — but produces audit noise + adds latency.

**Fix (pick one of three):**

- **(a)** Raise schema cap to 150 chars (cheap, doesn't hurt downstream — `summary_line` is already a "one-line summary" not a paragraph).
- **(b)** Move the constraint to the TOP of each agent's template (currently at the bottom under `## Output constraints`) so the LLM sees it before composing the JSON.
- **(c)** Post-process: truncate with ellipsis instead of rejecting. Costs strictness.

Recommendation: (a). Simplest, lowest behavior shift. Justified by real-task evidence that 100 chars is too tight for some legitimate single-line summaries on complex tasks.

**Test:** validator + reviewer schema test confirms 100-char input passes; 200-char fails; 150-char passes.

**Effort:** ~1-2h.
