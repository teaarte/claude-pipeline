# Phase v2.3 ŌĆö Daemon + Web UI + Multi-provider foundation

**Prerequisite:** v2 shipped (confirmed at commit `95f3f90`).
**Goal:** turn the in-process MCP-tool driver into a **long-running daemon** with HTTP API + minimal Web UI for configuration. Add the first non-shuttle `SpawnProviderPlugin` (Anthropic SDK direct) so model selection becomes meaningful. Keep Claude Code as a first-class entry point.

This phase is the bridge from "personal tool used in a Claude Code chat" to "self-hosted dev tool with multiple entry points and configurable LLM backends".

### What's already in place from v2 (don't redo)

These pieces were nudged into v2 ahead of time so v2.3 doesn't need rework:

- Ō£ō `ClaudePipelineConfig` type in `mcp/src/driver/types/config.ts` ŌĆö the schema Web UI will edit.
- Ō£ō `resolveAgentModel(plugin, phase, config)` cascade ŌĆö phase passed explicitly (no template_path heuristic).
- Ō£ō State IO encapsulated through `tools/*.ts` + `driver/core/state.ts` ŌĆö SQLite swap is a state-layer-only change.
- Ō£ō Driver transport-agnostic (`runFSM(state, registry)` accepts injected `SpawnRecorder`; HTTP API will inject its own).
- Ō£ō DriverŌåöpipeline-state fully wired via `mcpSpawnRecorder` ŌĆö open spawns close correctly through `pipelineBeginAgent` + `pipelineRecord*`.
- Ō£ō Concurrency-safe driver (`withDriverStateLock` on both `pipelineRunTask` and `pipelineContinueTask`).
- Ō£ō `pipeline_set_pattern_confidence` MCP tool (Item 11) ŌĆö past-misses confidence override already works.
- Ō£ō `pipeline_meta` MCP tool (Item 12) ŌĆö Web UI can call this to discover protocol version + tool list.
- Ō£ō `lib/ids.ts` consolidates id generators (don't write new ones; import).
- Ō£ō `lib/audit.ts` is lock-safe + bounded + redacted in global stream.
- Ō£ō `lib/project-dir.ts:assertProjectDirAllowed()` ŌĆö **MUST be used by HTTP API in v2.3.3 for every incoming `project_dir`** (Web UI is a path-traversal vector otherwise).
- Ō£ō Bypass marker is forgery-resistant (`issued_at + TTL cap` Ōēż 3600s) ŌĆö Web UI "Unlock writes" button calls existing `pipeline_unlock_writes`; do not reinvent the marker format.

v2.3 builds **on top of** these; reuse them, don't reinvent.

### Security must-haves carried over from v2

1. **HTTP API endpoints accepting `project_dir`** (POST /api/tasks, GET /api/tasks/:id, etc.): wrap every `project_dir` extraction through `assertProjectDirAllowed()` before passing to MCP tools. Without this, a malicious request can target paths outside the user's projects (e.g. `~/.ssh/`).
2. **Web UI "Unlock writes" button**: bound TTL to the same 3600s max enforced by `pipeline_unlock_writes`. Don't bypass.
3. **HTTP API task submission must use the SAME `mcpSpawnRecorder`** as MCP entry points ŌĆö guarantees pipeline-state stays consistent regardless of which client submitted.
4. **`INV_012` fires on both `completed` AND `skipped`** (review fix L3). If v2.3 adds gate-policy plugins that auto-skip phases, they MUST cancel open spawns first or hit this invariant.

### v2.3.1 ŌĆö Daemon lifecycle

- `claude-pipeline daemon start|stop|status|restart` CLI commands.
- PID file in `~/.claude-pipeline/daemon.pid`, log file in `~/.claude-pipeline/daemon.log`.
- Optional: `launchd` plist for macOS / `systemd` unit for Linux to autostart.
- Daemon process owns the singleton `PluginRegistry` + driver. All entry points connect to the same daemon.
- Health endpoint `GET /healthz` returns daemon uptime + plugin counts.

**Effort:** ~1 day.

### v2.3.2 ŌĆö SQLite migration for queryable + persistent project state

**Goal:** stop deleting per-task history at `pipeline_done_cleanup`. Mirror durable artifacts into SQLite tables, scoped per-project, so the daemon can answer "what happened on this project across all tasks" and the Web UI can render real dashboards. Working files in `<project>/.claude/` (plan.md, diff.txt, caller-context.md, etc.) still get cleaned ŌĆö only the **snapshots** persist.

This phase is load-bearing for everything downstream:
- v2.3 Web UI needs `tasks` + `findings` + `task_summary` to show non-empty dashboards.
- v2.6 curator agent cannot escalate-only-non-routine without `gate_decisions` + `findings` history to compare against.
- v2.5 cost-aware routing needs `spawn_costs` + `task_budgets`.
- Project changelog / memory / documentation auto-generation rides on `plans` + `task_summary`.

#### Project scoping

Every queryable table includes `project_id TEXT NOT NULL` + indexed. `project_id` is derived deterministically from the project's git-remote URL (preferred ŌĆö stable across machines) with fallback to canonical project_dir path. This lets the same machine host multiple projects in one DB without cross-contamination, AND eventually lets two machines (laptop + cloud) merge their tables along a stable project key.

Cross-project queries are first-class (single `WHERE project_id=?`); the schema is unified, not file-per-project.

#### `done_cleanup` becomes "snapshot + clean"

Today `pipeline_done_cleanup` deletes the per-task working files. In v2.3.2 it changes to:

1. **Snapshot phase** ŌĆö read state.gates / state.findings / plan.md / pipeline-state-summary.md / pipeline-state.json. Write rows into `tasks` / `findings` / `plans` / `task_summary` / `gate_decisions` in a single transaction.
2. **Clean phase** ŌĆö same as today: delete working files. `.claude/` returns to clean state for next task.

The snapshot is **idempotent** ŌĆö re-running `pipeline_done_cleanup` after a manual mid-task interruption MUST NOT double-insert. Use `INSERT OR IGNORE` on the unique task_id key, or check existence first.

#### Auto-derived artifacts (Web UI + curator unlock these)

With the schema above, the following become trivial queries ŌĆö no separate logic needed:

```sql
-- Project changelog over the last week.
SELECT date(ended_at) AS day, task_short, verdict, summary_md
FROM tasks t JOIN task_summary s USING(task_id)
WHERE project_id = ? AND ended_at >= date('now', '-7 days')
ORDER BY ended_at DESC;

-- Project's most-frequent finding categories ŌĆö "where this project tends to break".
SELECT category, COUNT(*) AS hits
FROM findings WHERE project_id = ? AND severity = 'blocking'
GROUP BY category ORDER BY 2 DESC LIMIT 10;

-- Reviewer accuracy: how often each reviewer's blockers were actually addressed.
SELECT agent, COUNT(*) AS total, SUM(CASE WHEN status='addressed' THEN 1 ELSE 0 END) AS addressed
FROM findings WHERE project_id = ?
GROUP BY agent ORDER BY total DESC;

-- Cross-task pattern mining: categories recurring across Ōēź3 tasks ŌåÆ candidate for new past-miss.
SELECT category, COUNT(DISTINCT task_id) AS task_count
FROM findings WHERE project_id = ? AND severity = 'blocking'
GROUP BY category HAVING task_count >= 3;

-- v2.6 curator: "has this gate-feedback pattern come up before?" ŌĆö match feedback fuzzy.
SELECT task_id, ts, feedback FROM gate_decisions
WHERE project_id = ? AND decision = 'reject' AND feedback LIKE ?
ORDER BY ts DESC LIMIT 10;
```

#### Retention policy

Storage grows linearly. Per 100 tasks Ōēł 3-5K findings + 100 plans + 100 summaries + 100-300 gate_decisions + 500-1000 spawn_costs. SQLite handles this without effort up to 10K+ tasks.

For projects running 5+ years, add a maintenance pass (CRON or manual): findings older than 12 months ŌåÆ keep aggregate per category, delete detail rows. Plans + task_summary keep everything (it's the changelog history; small per row).

#### Relationship with claude-mem (v2.2.5 Item 6 `MCPClientPlugin`)

claude-mem stores **observations** (semantic compressions of tool outputs). Pipeline DB stores **structured artifacts** (findings, plans, verdicts, decisions). These are complementary, not duplicative:

- claude-mem answers: "what context did the agent see while working" ŌåÆ fuzzy, semantic recall.
- Pipeline DB answers: "what did the pipeline conclude" ŌåÆ structured, queryable artifacts.

The Web UI shows pipeline DB content (clean tables); claude-mem stays as Claude Code-side plugin for context priming. No row-level overlap.

**Effort:** ~3 days. Schema design + migration scripts + `SqliteStateStorePlugin` implementation + `done_cleanup` snapshot transition + per-table indexes + smoke test on a project history of Ōēź10 tasks.

### v2.3.7 ŌĆö Permission strategy for autonomous mode

Claude Code asks the user for permission before running Bash commands, editing files, calling MCP tools, etc. In interactive `/task` flow this is fine ŌĆö user clicks through. In autonomous (daemon-submitted) tasks there's nobody to click. v2.3 must offer mechanisms that don't block on permission prompts.

**Background.** Claude Code permission system:
- `permissions.allow[]` in `~/.claude/settings.json` whitelists tools/commands.
- `defaultMode: "acceptEdits"` auto-approves file edits.
- `--dangerously-skip-permissions` CLI flag bypasses everything (used by ralphex).
- Task-spawned sub-agents inherit parent session's permission grants.

**Three strategies, configurable per task via `pipeline_config.permission_strategy`:**

#### Strategy B ŌĆö Claude Code subprocess + skip-permissions (DEFAULT for autonomous mode)

`ClaudeCodeSubprocessSpawnProvider` invokes `claude --dangerously-skip-permissions --output-format stream-json --verbose` per agent (ralphex pattern). Bypasses all permission prompts.

**Why this is the default:** v2.4 makes Docker isolation the default execution environment for autonomous tasks. With the container as the blast-radius boundary, `--dangerously-skip-permissions` is safe ŌĆö the agent can do whatever, but it can only do it inside the throwaway container. This combination = ralphex-grade autonomy + better isolation than ralphex (per-task containers vs ralphex's optional Docker wrapper).

**Pros:**
- Uses existing Claude Code subscription (no separate API key).
- Mirrors a battle-tested pattern (ralphex).
- Agent behavior is identical to interactive mode (no behavioral surprises).
- No permission-prompt deadlocks possible.

**Cons:**
- Requires `claude` CLI installed in the daemon's environment / Docker image (already true for our daemon container).
- "dangerously" is in the flag name ŌĆö pair with Docker isolation always.

**Effort:** ~1 day, slots into v2.3.4 work.

#### Strategy A ŌĆö Anthropic SDK direct (alternative for headless or API-first setups)

`AnthropicSdkSpawnProvider` calls `@anthropic-ai/sdk` directly. Claude Code permission system never engages because we're not using Claude Code at all for the spawn.

Tool surface is explicit in the SDK call:

```typescript

  model: agentConfig.model,
  tools: BUILTIN_AGENT_TOOLS,  // Read, Edit, Bash (with shell filter), Grep, etc.
  messages: [...],

```

**When to use:** running daemon on a server without Claude Code CLI installed; CI integrations; cost monitoring through Anthropic console rather than Claude Code subscription; needs explicit per-tool audit.

**Pros:** explicit tool surface, no Claude Code CLI dependency, separate Anthropic billing visibility, foundation for multi-model providers in P5.
**Cons:** requires `ANTHROPIC_API_KEY`; separate billing from Claude Code subscription; agent tools must be defined explicitly (more work than just inheriting Claude Code's defaults).
**Effort:** already counted in v2.3.4.

#### Strategy C ŌĆö Pre-warmed allowlist + shuttle (alternative for paranoid users)

For users who want subprocess mode WITHOUT `--dangerously-skip-permissions` even with Docker isolation: daemon generates `~/.claude-pipeline/auto-settings.json` per task with computed allowlist and spawns Claude Code subprocess with `--settings <path>`.

**When to use:** belt-and-suspenders security in environments where Docker isolation is considered insufficient (e.g., shared CI runners).

**Pros:** controlled blast radius even inside the container.
**Cons:** allowlist is a guess; tasks needing unexpected commands halt; complexity.
**Effort:** ~1-2 days, can be added later if A+B insufficient.

### v2.3.8 ŌĆö Branch isolation + merge strategy (worktree + auto-merge)

Each autonomous task runs in a `git worktree` isolated from the main working tree:

```
<repo>/
  .git/                                    # main git dir
  src/                                     # main branch checkout
  .claude-pipeline/worktrees/
    t-2026-05-13-feature-x/                # worktree for autonomous task
      src/                                 # isolated checkout of task branch
      .claude/                             # task-local state
```

`pipeline_init` accepts `isolation: "worktree" | "in-place"` (default `in-place` for interactive Claude Code `/task`; default `worktree` for autonomous Web UI / CLI submissions).

#### Branch strategy (where task runs)

| Strategy | When | Behavior |
|----------|------|----------|
| `in-place` | Interactive `/task` in Claude Code chat (default) | Task runs on current branch. User sees changes immediately. No worktree. |
| `new-branch` | Autonomous Web UI submission (default) | Daemon creates `claude-pipeline/<task_id>` branch from configured base. Worktree checks it out. All commits land on this branch. |
| `existing-branch` | Submitter specifies target branch in submit form | Like `new-branch` but checks out an existing branch (e.g. resume work on a feature branch). |

Branch name pattern is configurable: `branch_name_template: "claude-pipeline/{task_id}"` (defaults shown; can be `"feat/{task_short}"` or whatever).

#### Merge strategy (what happens when task completes successfully)

User-selectable in **Web UI submit form** (overrides global default in Settings):

| Strategy | Behavior | When to use |
|----------|----------|-------------|
| `no-merge` (manual) | Task branch left untouched. UI shows "Merge ready" with link to open PR or merge locally. | Code review desired before integration. Default for first-time users. |
| `auto-merge` | After `pipeline_finish` succeeds, daemon `git merge --no-ff <task-branch>` into base (preserves task history). | Trusted autonomous flows where review already happened via in-pipeline reviewers. |
| `auto-squash-merge` | Same as `auto-merge` but `git merge --squash` + auto-commit with summary message. Loses individual task commits, keeps single "feat: <task description>" commit. | Clean linear history preferred. **Toggle in UI** as per user request. |
| `auto-rebase-merge` | `git rebase` task branch onto base, then fast-forward. | Linear history without explicit merge commits. |

#### Auto-merge safety preconditions (HARD)

Auto-merge ONLY proceeds when ALL of:

1. `pipeline_finish` returned successfully (`verdict: "accepted"`, no INV violations).
2. All gates were either approved or auto-approved per `GatePolicyPlugin`.
3. All tests in `phases.test_first.test_files_written` and `phases.validation` are green.
4. No `pipeline_violation` flag is set on state.
5. `git merge --no-commit` dry-run shows no conflicts with base.

If ANY of these fails ŌåÆ fall back to `no-merge` (manual), with notification explaining which precondition blocked.

Conflict on attempted merge ŌåÆ daemon aborts the merge cleanly (`git merge --abort`), leaves the task branch, and emits a `merge-conflict` notification with the conflict file list. User resolves manually.

#### Per-merge audit

Every auto-merge attempt (success OR failure) appends to `~/.claude/metrics/mcp-audit.jsonl` via the existing `audit()` helper:

```json

  "schema_version": "1.0",
  "ts": "...",
  "tool": "branch:auto-merge",
  "task_id": "t-...",
  "args_summary": {
    "branch": "claude-pipeline/t-...",
    "base": "main",
    "strategy": "squash-merge",
```

This gives `/learn` data about which merge strategies users prefer + how often auto-merge gets blocked by preconditions.
