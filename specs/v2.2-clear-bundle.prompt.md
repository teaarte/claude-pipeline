- **Q28 + Q29 + Q37 first** (~1.5h): mechanical prompt / vocab / one-line extraction edits. Build momentum, expand test surface, no risk.
- **Q32 + Q34 + Q31 + Q10** (~1h): four schema deprecations. Doing all four together = one mental context (`templates/schemas/pipeline-state.schema.json`); reduces context-switching.
- **Q33** (~1h): biggest mechanical change — new helper file (`lib/git-diff.ts`), integration test. Place after schema-deprecation because Q33 might naturally adjust the schema (adds `deleted` field?).
- **Q26** (~1-2h): touches the v2.1 Q17 detector — needs careful priority-chain logic. Place when context is freshest.
- **Q25** (~30min): docs-only. Place last as a cool-down item.

## Why this bundle, not the others

- **Q9 + Q27 + Q30 deferred to v2.2a** — these need a fresh real-task run (preferably auth/perf-touched, to trigger all 5 reviewers via `applies_to`) to verify the fan-out + input-files + ref-injection actually work end-to-end. Unit tests alone can't tell you that "the review surface is firing correctly" — only a real-task can.
- **Q1-Q6 deferred to v2.2-code-polish** — code-quality items (33 `any` types, hot file refactor, output-example DRY) are independent of validation findings and benefit from a separate review pass.

## Risk register

- **Schema deprecation breaks old `pipeline-state.json` fixtures.** Mitigation: ajv `additionalProperties: true` policy lets old fixtures still validate; new state generation just won't write those fields. Verify the policy before shipping Q31/Q32/Q34/Q10. If `additionalProperties: false` is in effect, add a migration test fixture.
- **Q33 git-diff dependency** could break in worktrees without git, monorepos with subdir initialization, etc. Mitigation: graceful empty-fallback + audit log. Don't crash.
- **Q26 monorepo classification** is a judgment call — adding `"monorepo"` as a 4th `project_type` requires touching consumers. If touching > 3 files, just stay with `"library"` and file Q-followup for proper monorepo handling.

## Why NOT use `/task` to dogfood this bundle

1. **Pipeline review surface is currently incomplete** (Q9, Q27, Q30 open). On code-quality changes, `/task` would spawn only logic-reviewer (Q9), reviewing blind to diff.txt / caller-context / past-misses (Q27), without injected domain refs (Q30). You'd get LESS review value than just executing systematically via this launcher.
2. **Self-referential dogfooding pollutes state.** `claude-pipeline` repo's own `.claude/` becomes ambiguous: does it belong to the project being orchestrated, or to the orchestrator running on the project? Filename collisions + namespace mixing.
3. **Mechanical fixes don't benefit from adversarial review.** These are schema deletions, vocab additions, one-line extractions, docs. Logic-reviewer would find ~0 blockers; acceptance is just lint/typecheck/test/build. Pipeline overhead (~8x token cost) without proportional value.

**The right dogfood moment:** after v2.2a closes (Q9 + Q27 + Q30 fixed). Then run `/task` on a separate real backend/frontend project, observe whether the multi-reviewer fan-out actually fires with full context. That's the validation worth doing.
