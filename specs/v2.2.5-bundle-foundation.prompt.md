### Item 1 — Phase as `string` (not enum)

**Symptom:** `Phase` is hard-coded enum in `mcp/src/lib/phase-state-machine.ts` with 6 values (`context | planning | test_first | implementation | validation | final`). Adding a phase for a future TikTok bundle (e.g. `draft`, `render`, `publish`) would require touching this enum + every consumer. Goal: flow declares its own phases as `string[]`; type system treats Phase as `string`.

**Fix:**

1. In `mcp/src/lib/phase-state-machine.ts`: change `Phase` from enum to `type Phase = string`. Add export `CODE_PHASES: readonly string[]` with current 6 values for backward-compat callers that need the list.
2. In `mcp/src/driver/types/plugin.ts`: `FlowPlugin` interface gains `phases: string[]`.
3. In current built-in flows (`mcp/src/driver/builtin/flows/{simple,medium,complex}.ts`): each declares `phases: [...CODE_PHASES]` (or the subset relevant to that flow's complexity).
4. State schema (`templates/schemas/pipeline-state.schema.json`): `phases` keys validation moves from ajv enum to runtime check in `pipeline_validate` — accept any string key, but if any verdict references a phase, the FSM checks against `flow.phases`. (Or keep ajv but make additionalProperties: true under `phases`.)
5. `phase-state-machine.ts` `isValidPhase()` helper: accept string + active flow's `phases[]`.
6. Grep for `Phase.context`, `Phase.planning`, etc. usages — replace with string literals or constant from `CODE_PHASES`.

**Tests:**
- `mcp/test/lib/phase-state-machine.test.ts` — existing tests still pass with Phase=string.
- New test: a synthetic flow declares `phases: ["alpha", "beta"]` — state with `phases.alpha.status` validates clean against schema; transition from alpha to beta works via state machine.
- New test: state with phase name not in active flow's phases → `pipeline_validate` flags as INV_011-style violation.

**Acceptance:** all 343 existing tests pass + ≥3 new tests covering generic phase names. Typecheck clean. No hardcoded `Phase.X` references remain in `core/`.

**Commit subject:** `refactor(types): item 1 — Phase as string-typed; FlowPlugin declares phases[]`

### Item 2 — `state.bundle` field + schema split

**Symptom:** Pipeline state schema is monolithic. All fields (code-specific like `tests_mode` / `stack` and universal like `verdict` / `gates` / `reviewer_verdicts`) live in one file. Adding a TikTok bundle would require either adding `content_type` / `audience_signal` etc. directly to the base schema (polluting), or fork+rewrite (untenable).

**Fix:**

1. Bump `schema_version` to `"1.1"` in `templates/schemas/pipeline-state.schema.json`.
2. Add `bundle: { type: "string", default: "code" }` as a top-level required field (with default value — old state files without it still validate). Or use ajv `useDefaults: true` and mark not-required.
3. Split schema into two files:
   - `templates/schemas/pipeline-state.schema.json` — base (universal: task_id, complexity, verdict, gates, reviewer_verdicts, phases, findings_path, agents_count, bundle, started_at).
   - `templates/schemas/bundle-extensions/code.schema.json` — code-specific (tests_mode, stack, files, blockers_found, reviewer_misses_post_merge, plan_iters, impl_iters, gate1_revisions). These fields become required ONLY when `bundle === "code"`.
4. ajv conditional validation in `mcp/src/lib/schemas.ts` (or wherever): load base + bundle's extension; combine and validate.
5. `pipelineInit` sets `bundle: "code"` automatically if no `pipeline.config.json` present (legacy projects continue to work).
6. Update `templates/pipeline-state.json` template (the initial state) to include `bundle: "code"`.

**Tests:**
- Existing tests use the new combined schema; should pass unchanged.
- New test: state with `bundle: "code"` + missing `tests_mode` → invalidates per code-extension.
- New test: state with `bundle: "tiktok"` (synthetic) + no `tests_mode` → validates fine (tests_mode not required for tiktok).
- New test: legacy state without `bundle` field → `pipeline_validate` accepts (default fills in via ajv) AND assigns `code`.
- New test: schema_version 1.0 vs 1.1 backward compatibility — old fixture files still validate.

**Acceptance:** all tests pass + ≥4 new ones. Old state files (v2.x without `bundle` field) silently migrate to `code` bundle.

**Commit subject:** `feat(schema): item 2 — add state.bundle field + split schema into base + per-bundle extensions`

### Item 6 — `MCPClientPlugin` contract + spawn lifecycle

**Symptom:** Pipeline is an MCP server today. To integrate other MCP servers (claude-mem for memory, search MCP, github MCP, etc.) requires manual external setup. Goal: declarative integration — config-level addition of external MCP servers; pipeline daemon spawns them and exposes their tools to agents.

**Fix:**

1. New plugin contract in `mcp/src/driver/types/plugin.ts`:
   ```typescript
   export interface MCPClientPlugin extends PluginMeta {
     name: string;
     server_command: string[];
     env?: Record<string, string>;
     expose_tools: string[];
     scope: "task" | "team" | "global";
     health_check?: { tool: string; timeout_ms: number };
   }
   ```
2. New file `mcp/src/driver/bundles/code/mcp-clients/` — empty for now (no required external MCP servers). Document the directory for future bundles.
3. New file `mcp/src/lib/mcp-client-manager.ts`:
   - `spawnExternalServer(plugin: MCPClientPlugin): Promise<ChildProcess>` — fork/spawn the server command.
   - `handshake(child: ChildProcess, plugin: MCPClientPlugin): Promise<MCPCapabilities>` — perform MCP protocol handshake.
   - `getExposedTools(): Tool[]` — return tools from all spawned external servers.
   - `lifecycle hooks` — spawn on task start (scope: task), keep alive between tasks (scope: team/global).
4. `AgentPlugin` gains optional `mcp_tools?: string[]` field — which external tools this agent is allowed to call. Validates against MCPClientPlugin's `expose_tools`.
5. `pipeline.config.json` extension:
   ```json
   {
     "mcp_clients": [
       {
         "name": "claude-mem",
         "server_command": ["npx", "claude-mem", "mcp-server"],
         "expose_tools": ["search", "timeline", "get_observations"],
         "scope": "team"
       }
     ]
   }
   ```
6. `pipeline_init` reads config, spawns MCPClientPlugins of scope `team`/`global` at daemon start (or task-start for `task` scope when not in daemon mode). Audit log entry for each spawn.
7. **No actual claude-mem integration in this commit.** Just contract + spawn lifecycle + one MOCKED external MCP test to prove the pattern works.

**Tests:**
- `mcp/test/lib/mcp-client-manager.test.ts` — new file.

- Negative: external server fails to spawn → audit entry, pipeline continues (graceful degrade).

## Bundle order rationale

- **Item 1 (Phase as string) first** — foundation type-system change. Many downstream items depend on this generalization. Doing it first means everything else inherits the cleaner abstraction.
- **Item 2 (schema split)** — depends on Phase being flexible; also a type-level change.
- **Item 3 (directory move)** — mechanically large but architecturally simple. Doing it after type changes means imports get fixed in one pass.
- **Item 4 (BundleManifest + loader)** — depends on directory structure (item 3) + types (items 1-2). Closes Q40.
- **Item 5 (skills)** — depends on bundle existence (item 4) for `task-prompt.md` resolution.
- **Item 6 (MCPClientPlugin)** — independent type/contract addition; place where context is fresh.
- **Item 7 (team knowledge)** — simplest, ends bundle on a light note.

## Why this bundle, not larger

- **No actual second domain bundle authored.** `_template/` skeleton only — premature to build TikTok/marketing/research without validated demand.
- **No write API for team knowledge.** v2.6 curator decides promotion logic.
- **No `pipeline.config.json` editor UI.** v2.3 Web UI delivers that.
- **No MCP marketplace.** Config-level integration is enough for now.
- **No LLM-classification activation.** That's v2.3 daemon's non-shuttle SpawnProvider work.

v2.2.5 is **substrate only**. Every future phase (v2.3+) builds on top with confidence that the bundle abstraction is real, not a Q40-deferred future promise.
