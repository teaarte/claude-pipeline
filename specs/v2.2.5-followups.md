## 🔴 HIGH — must close before real-task v2.2.5 verification

### H1. `continue-task.ts` zod schema for `user-answer` doesn't match runtime contract (regression from Item 8)

**Location:** `mcp/src/driver/tools/continue-task.ts:48-52` (look for the `user-answer` branch of the zod union for ContinueTaskInput).

**Symptom:** Item 8 switched the type and handler to `{decision: "accept" | "reject", message?: string}`, but the zod input schema still requires `answer: z.string()`. Any structured-shape MCP call is rejected at the zod boundary; any legacy `{answer: "..."}` call passes zod but writes `{decision: undefined, message: undefined}` into scratch.

**Fix:** Replace the `user-answer` zod branch with:
```typescript
z.object({
  driver_state_id: z.string(),
  type: z.literal("user-answer"),
  decision: z.enum(["accept", "reject"]),
  message: z.string().optional(),
})
```
No legacy `answer` field. Cross-check that no test, smoke harness, or skill markdown still passes `answer: ...`.

**Test:** integration-level test in `mcp/test/driver/tools/` or extend an existing `gate-mirror.test.ts` case to call `pipelineContinueTask` with the structured shape and assert `state.gates.gate0 === "approved"`.

### H2. `team_knowledge` dropped on multi-agent REVIEW fan-out (regression from Item 7)

**Location:** `mcp/src/driver/bundles/code/steps/index.ts` around the `REVIEW` step's `provider.spawn({...})` calls (~lines 447-455 per the review). Look for the `spawnAgentsParallel` orchestration or direct `provider.spawn` calls inside the REVIEW StepPlugin.

**Symptom:** `spawnOne(...)` correctly resolves `team_knowledge` via `loadTeamKnowledgeForSpawn(state)`. The REVIEW step's fan-out invokes `provider.spawn` directly without that field, so logic/challenger/style/security/performance reviewers never see team knowledge — Item 7 broken on the largest spawn surface.

**Fix:** Hoist `const teamKnowledge = await loadTeamKnowledgeForSpawn(state)` to the top of the REVIEW step (above the fan-out), pass `team_knowledge: teamKnowledge` to every `provider.spawn(...)` request in that fan-out. Also audit any other multi-spawn step (`PLAN_REVIEW`, `ITERATE`, etc.) for the same omission.

**Test:** Extend `mcp/test/driver/bundles/code/spawn/shuttle-provider.test.ts` OR add a step-level test to assert that when `state.team_knowledge_refs` is set + tempProject has the file, the spawn provider receives the content in every fan-out request.

### H7. `pipeline-guard.sh` false-positive: `bash -c "cat …"` denies legitimate reads

**Location:** `hooks/pipeline-guard.sh:49` (WRITE_OP_PATTERNS).

**Symptom:** The pattern `(^|[[:space:]])(bash|sh|zsh)[[:space:]]+-c[[:space:]]` marks ANY `bash -c "..."` invocation as a write-op, including pure reads (`bash -c "cat .claude/pipeline-state.json"`). Hooks/README.md explicitly promises reads pass through.

**Fix (chosen):** Keep the `bash -c` pattern recognition but inspect its body for an inner mutator before classifying as write-op. Extract the body via a bash function (e.g. `extract_shell_body "$cmd"` that grabs the quoted argument after `-c`), then check the body against an explicit mutator list: `rm `, `mv `, `cp `, `>`, `>>`, `truncate`, `dd if=`, `dd of=`, `eval`, `perl -e`, `python -c`, `node -e`. If the body matches none of these, treat as read → pass through.

**Test:** Add a test case under `tests/guard-evasion/` (or wherever the bash hook tests live) — `bash -c "cat .claude/pipeline-state.json"` should pass; `bash -c "rm .claude/pipeline-state.json"` should deny.

### H8. `mcp-client-manager.ts` unhandled child_process `error` event

**Location:** `mcp/src/lib/mcp-client-manager.ts:142-150` (the default `transport.spawn` implementation).

**Symptom:** `nodeSpawn(cmd, args, ...)` with an invalid `cmd` (ENOENT) emits an `error` event asynchronously. No listener is attached → unhandled `'error'` event crashes the whole MCP server process.

**Fix:** Attach `proc.on('error', ...)` immediately after `nodeSpawn`. Race the `spawn` event vs first `error` event before resolving the spawn promise; on error, throw or return a rejected promise that the manager's existing `try/catch` will turn into the `mcp-client-spawn-failed` audit entry.

**Test:** Extend `mcp/test/lib/mcp-client-manager.test.ts` with a real (not synthetic-transport) test using `server_command: ["nonexistent-binary"]` — assert manager catches gracefully and emits the spawn-failed audit, no process crash.

### H9. `mcp-client-manager.ts withTimeout` leaks child process on timeout

**Location:** `mcp/src/lib/mcp-client-manager.ts:159-174` (the `withTimeout` helper).

**Symptom:** On timeout, the wrapper rejects, but the underlying `transport.spawn(plugin)` promise keeps running. When it eventually resolves, the spawned ChildProcess has no consumer → orphan.

**Fix:** Pass an `AbortSignal` (or a cleanup hook) into `transport.spawn` so the timeout path can call `kill()` on the late-resolving child. The default transport should accept the signal and abort the spawn. For synthetic transports in tests, the cleanup hook is a no-op.

**Test:** Hard to write deterministically without injecting timing — at minimum, add a test that uses a slow synthetic transport (`new Promise(resolve => setTimeout(resolve, 100))`) with `health_check.timeout_ms: 10`, asserts the manager records the timeout, and that the cleanup hook was invoked.

### H10. `set-pattern-confidence.ts` JSONL rewrite has no lock vs concurrent `log-agent-feedback`

**Location:** `mcp/src/tools/set-pattern-confidence.ts:30-53`.

**Symptom:** Race window — between `readFile` and `rename(tmp, file)`, another tool can append a feedback entry to the same JSONL. The set-pattern-confidence rewrite then clobbers the appended entry.

**Fix:** Wrap the read-modify-rename in `lockfile.lock(agentFeedbackJsonl, …)`. Ensure `appendJsonl` (or wherever `log-agent-feedback` writes) uses the SAME lock. Easiest: extract a shared `withFeedbackLock(fn)` helper into `lib/state-io.ts` and use it from both call sites.

**Test:** Concurrent test — two simultaneous calls (one `log-agent-feedback`, one `set-pattern-confidence`), assert both entries are preserved.

### H11. `audit.ts appendCapped` fast-path race exceeds cap

**Location:** `mcp/src/lib/audit.ts:127-170`.

**Symptom:** Two concurrent `audit()` calls can both observe `st.size <= cap * 300` on the unlocked fast path, both append, regularly exceeding cap.

**Fix:** Choose one:
- **(a) Always lock** — drop the fast-path, every append goes through the lock. Simpler, slightly slower under steady state.
- **(b) Accept 2× drift, document** — add a comment noting the cap is a soft limit (~cap, not strict ≤ cap).

Recommend (a) for predictable storage behavior — the audit log isn't latency-critical.

**Test:** Two concurrent `audit()` calls into a near-cap file (priming with `cap * 300` size beforehand); assert no entry loss AND file size stays within `cap * 350` after both complete.

### H12. `state-io.ts withStateLock` silently drops mutations on empty-state files

**Location:** `mcp/src/lib/state-io.ts:60`.

**Symptom:** `existing && Object.keys(existing).length > 0 ? existing : null` returns `null` for an empty-object state file. If the callback does `if (state) state.x = ...`, the assignment skips entirely; the lock release writes nothing — caller thinks success, disk is unchanged.

**Fix:** Either:
- Validate by reading `schema_version` AND `task_id`/`bundle` fields. If they exist, treat as real state. If file exists but is missing required fields, throw a clear error (`CORRUPT_STATE`) instead of returning null.
- OR drop the empty-object special-case entirely — let callers handle `state === undefined` (genuinely no file) vs `state.task_id === undefined` (corrupt) themselves.

The first is safer. Pick that.

**Test:** Write `{}` to a state file path, call `withStateLock(file, fn)` — assert `fn` either receives the empty object (so it can detect corruption) or throws a clear `CORRUPT_STATE` error. NOT a silent null.

### H13. `fix-task-id.ts` doesn't update findings.jsonl / reviewer_verdicts

**Location:** `mcp/src/tools/fix-task-id.ts:34-64`.

**Symptom:** Rewrites `state.task_id` only. findings.jsonl already on disk carry the OLD id. Cross-task analysis joining findings → metrics by task_id finds zero matches.

**Fix (chosen):** Atomic rewrite. In the same lock, read findings.jsonl, rewrite every entry's `task_id` field where it matches the OLD id, write back via tmp + rename. Steps:
1. Acquire state lock (already exists in the tool).
2. Acquire findings.jsonl lock (new — coordinate with `record-agent-run.ts` writers).
3. Read findings.jsonl, parse each line, swap `task_id` where appropriate, write to tmp, rename.
4. Update `state.task_id` (existing behavior).
5. Release both locks.

If findings.jsonl read or write fails, throw before mutating state.task_id — caller can retry.

**Test:** Init a task with task_id X, record a finding, call `pipelineFixTaskId({old_task_id: X, new_task_id: Y})`, then read findings.jsonl and assert the entry's task_id is Y. Also assert atomic behavior — if the rewrite is interrupted mid-write, state.task_id remains X.
