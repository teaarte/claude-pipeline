## Boundary notes

1. **CC-binding scope contracted.** D4 renamed `claude_code_task` → `spawn_request` with a runner-agnostic `runner_hint` field. `mcp/src/driver/types/shuttle.ts` no longer mentions Claude Code in its type names. The shuttle provider sets `runner_hint: "claude-code-task"`; the skill markdown reads the hint to branch the harness invocation. Future Cursor / Anthropic SDK / daemon adapters declare their own hint values without touching pipeline core types.

2. **Classifier failure mode is informational, not load-bearing.** D1's `extract-classifier-output` hook validates against the schema; bad output keeps existing defaults intact and audits `llm-classification-needed`. The pure-getter decisions (`refs-to-load`, `security-needed`, `antipattern-rules-applicable`) return empty/false defaults when the slot is unset — the pipeline never blocks on a classifier hiccup. This preserves the design: classifier is an LLM-tool optimization, not a workflow gate.

3. **`auto_replan_on_blocking_max` defaults to 0.** No behavior change for users who don't opt in. D9 is observable only via the `pipeline.config.json` field + audit rows.

4. **D4 audit row addition.** Every spawn through `shuttleSpawnProvider` now emits a `pipeline_spawn` audit entry with `runner_hint` in args_summary. Pre-D4 spawns did not audit at the provider level (only the begin-agent layer logged via `pipeline_begin_agent`). Cross-task analytics can now answer "what % of spawns went through which adapter?".
